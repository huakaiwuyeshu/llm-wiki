## Part 1: 什么是 Agent + Agent vs Workflow

**核心要点**:
1. **Agent 的四要素**(1 段):环境/传感器/执行器/**自主性** ← 自主性是核心
2. **自主性是分水岭**(2 段,张力叙事第一拍):
   - Workflow = 按部就班执行指令(流程写死)
   - Agent = 被赋予自由度自主达成目标(动态推理决策)
   - 金句:"Workflow 是让 AI 按部就班地执行指令,而 Agent 则是赋予 AI 自由度去自主达成目标"

## Part 2: Agent 怎么工作 · Agent Loop

**核心要点**:
1. **循环是什么**(1 段):Perception → Thought(想)→ Action(做)→ Observation(看)→ 闭环迭代
2. **金句**(highlight 框):"Agent 不是一种产品形态,而是一种控制策略——就是这个循环本身"
3. **业务例子走一遍**(1 段):用「三方说签名失败」走一轮循环
   - Thought:"需要 appid、接口路径、签名参数、错误码"
   - Action:追问用户
   - Observation:用户提供日志
   - Thought:"签名参数顺序错误"
   - Action:给出排查建议
4. **生产实践**(1 段):90% 场景用「ReAct 主循环 + 按需 Plan」,能不规划就不规划

## Part 3: 企业落地的关键认知 · 为什么收敛成混合架构

**核心要点**(张力叙事第二拍):
1. **承上启下钩子**(1 段):"既然 Agent 和 Workflow 有本质区别(自主性),为什么企业落地后两者又长得像了?"
2. **控制权钟摆**(2 段):
   - 早期:全用规则 → 死板
   - 中期:全交 LLM 自主 → 灵活但失控("业务最怕的不是系统不够聪明,而是系统自作聪明")
   - 收敛:工程实践把控制权收回来 → **「Workflow 负责边界,Agent 负责弹性」**
3. **一句话收束**(highlight):"企业业务 Agent 不是'更会聊天的机器人',而是一个承接高频业务流程的**工作系统**——LLM 只负责理解与表达增强,由 Workflow、Schema、Skill、Task State、Memory、Audit 和人工审核共同负责稳定、可控、可复盘"
4. **为什么选混合架构**(1 段 + 表格):业务特征 → Agent 需要什么 → 对应模块
   - 流程顺序稳定 → 不要每次重想流程 → Workflow
   - 字段规则明确 → 判断可校验 → Schema + Validator
   - 问题高度重复 → 复用专家经验 → Skill + 知识库
   - 任务多轮推进 → 知道走到哪一步 → Task State
   - 凭证上线敏感 → 审计与人工确认 → Audit + 人工审核
5. **判断标准**(1 段):不是"回答得像不像",而是"能不能按流程稳定推进"

## Part 4: Agent 演进 · 怎么长成今天这样

**核心要点**:
1. **导语**(1 段):"在进入具体模块前,先看 Agent 是怎么一步步演进到今天的混合架构形态"
2. **四阶段演进 · 自主度视角**(1 段 + 图):早期 Agent(聊天+工具调用)→ 工作流 Agent(流程编排)→ 自主 Agent(规划+执行)→ 自进化 Agent(Skill+Memory 沉淀经验);**启发**:不是越自主越好

## Part 5: 各模块解析 · 混合架构怎么搭

**性质**:手册核心,理解成一套业务 Agent 的标准工作分工  
**结构重排**:按主链路「用户输入 → 会话路由 → 需求转化 → Schema 校验 → 调用 Skill → 输出确认」顺序讲,每个模块挂在所在节点

### 5.0 主链路全景(1 段 + 图)
- **业务主流程**:用户输入 → Session Router → 需求转化 → Schema 校验 → 调用 Skill → 输出确认
- **LLM 增强层**:挂在多个节点(识别意图、抽取字段、追问润色、输出润色),LLM 是增强点,不是唯一决策者
- **支撑与边界四件套**:Task State(记住进度)、Memory(沉淀经验)、Audit(记录风险)、人工确认(高风险放行)
- **Workflow 归位说明**(新增 highlight 框):"Workflow 是贯穿全链路的骨架,定义「哪些步骤必须走、哪些权限不能越、哪些结果必须审核」,不单列为模块编号,而是体现在主链路的流程顺序里"
- **LLM 放在哪里**(表格):Session Router / 需求识别 / 追问 / 知识问答 / 输出润色 × (适合做什么 vs 不应独自做什么)

**字数预估**:600 字  
**图表**:图 5(修正原手册图 4):主链路 + 增强层 + 四件套(**修掉编号 bug,原 p4 标"图4",p9 也标"图4"**)

### 5.1 主链路模块(按数据流顺序)
1. **入口层**:WebUI/CLI/未来本地接口;价值是把业务需求自然地接进 Agent
2. **Session Router(会话分诊台)**:判断闲聊/新任务/补充信息/修改任务;避免"好的""这个不对"都当新任务
3. **需求识别与转化**:NL → 结构化任务(如"签名失败" → `request_type=signature_debug`);LLM 帮助理解,不负责最终放行
4. **Schema Builder + Validator**:Schema 是任务单模板,Validator 是检查员;三态 `executable` / `needs_clarification` / `blocked`;**金句**:"LLM 可以帮你填表,但不能自己决定表是否合格"
5. **Clarification Policy(追问策略)**:缺什么只问什么;四类(缺失/模糊/冲突/非法);问哪些字段由 Schema 决定,LLM 只做话术润色
6. **Skill Router + SkillLoader**:Skill Router 像派单系统(新接入→接入流程 Skill,签名失败→鉴权排查 Skill);SkillLoader 渐进式加载(Advertise → Load → Read → Run)
7. **知识库与文档检索**:先检索文档再回答,无依据明说未找到;回答可追溯、口径一致
8. **Tool Governance(工具治理)**:管理工具能不能调用、谁有权限、是否 dry-run、是否需人工确认、是否记审计日志;一期轻量,接真实系统前加强

**字数预估**:2000-2500 字(3.5 页)

### 5.2 支撑与边界模块
9. **Task State**:记住任务状态(需求确认中/资料待补充/凭证待生成/沙箱联调中/上线前检查/已上线);**解决**:多轮推进,不是每轮从零开始
10. **Memory**:记录识别错误、用户纠正、真实联调案例、重复问题、Skill 优化建议;**解决**:"把个体经验逐步变成组织能力"
11. **Runtime Logs / Audit**:记录用户输入摘要、识别成什么、Schema 结果、命中哪个 Skill、是否触发安全边界;**但不能记敏感信息**(appsecret、密钥)
12. **人工审核发布**:最后一道边界;Agent 可生成建议/草稿/排查步骤,**不能自动**生成 appsecret、发放凭证、改后台配置、上线、更新生产 Skill;**金句**:"Agent 是帮人推进流程,不是绕过人和权限系统"

**字数预估**:1200-1500 字(2 页)

### 5.3 一句话速记(highlight 框)
"Workflow 管流程,Schema 管判断,Clarification Policy 管追问,Skill 管能力,Task State 管推进,Memory 管进化,Audit 和人工审核管风险,LLM 负责理解和表达增强"

**字数预估**:简版 800 字 → 详版 4000 字(6 页)  
**素材来源**:原手册 p4-7(笔记已全文蒸馏) + `wiki/concepts/enterprise-agent.md`(四、五节)

## Part 7: API 接入与分析 Agent · 实战分析

**核心要点**:
1. **为什么 API 接入适合第一批业务 Agent**(1 段):固定流程、固定材料、固定文档、固定排查套路、明确安全边界;三方常见问题大量重复
2. **三条任务链**(1 段 + 图):新 API 接入(需求确认→资料补齐→凭证准备→上线检查)、联调问题(识别→补日志→排查建议→沉淀案例)、单量分析(确认口径→趋势分析→异常判断→输出周报);共用底座 Schema/Skill/Task State/Memory/Audit
3. **端到端案例 · 三方说"签名失败"**(表格,复用原手册 p10):8 个阶段(入口层→会话路由→需求识别→Schema 校验→追问→Skill 路由→输出→Memory/Audit)× (Agent 做什么 + 产品经理关注什么)
4. **安全边界红线**(highlight 框):"appsecret 不进模型上下文、不在对话中复述、不写入普通日志;查看/重置/发放必须走有权限和审计的安全通道"
5. **代码参考 · 从开源模板学设计**(2 段,新增):
   - **trust-gated 信任闸门**(来自 awesome-llm-apps):多 agent 流水线,每步参与前先打分(0-100 金/银/铜分级),不达标直接挡;SHA-256 哈希链审计日志,任一条篡改后面全对不上 → 对应 Tool Governance + 人工审核"达标才放行"
   - **deterministic-picker 思路**(来自 rag_failure_diagnostics_clinic):把判断交给代码而非 LLM——预定义 12 种失败模式枚举,System Prompt 硬写死"只能从枚举里选,不许发明新 id" → 对应 Schema 需求识别的"LLM 帮你填表,代码决定合不合格"
   - **统一抄法三问**:① 把"判断"交给 LLM 还是代码?② 兜底怎么处理?③ 沉淀了什么可复盘记录?
6. **一期能力范围**(列表,复用原手册 p9):接住新接入诉求、回答 API 文档常见问题、追问关键日志给排查路径、样例数据做单量分析、记录重复问题形成优化建议

## Part 8: 结语 · 业务 Agent 的目标

**核心要点**(1 段):
"业务 Agent 的目标不是替代所有人,而是把重复流程变成可复用能力,把个人经验变成组织资产,把大模型的表达能力放进可控的业务系统里。当一个业务 Agent 能稳定处理核心场景,且日志、Memory、Skill 更新机制跑通后,再抽象成通用底座;此时新业务只需替换 Workflow、Schema、Skill、知识库和验收集,不用重新发明整个 Agent 架构。"

**字数预估**:200-300 字(0.5 页)
