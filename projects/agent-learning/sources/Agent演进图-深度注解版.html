内部学习稿 · 给产品经理的 Agent 演进课

Agent 演进图 · 深度注解版

每一步演进,都是被「上一代的痛」逼出来的补丁。读法:上一代缺陷 → 新设计 → 为什么非这么设计不可。九个阶段,可以归到三个递进的大问题里。

总纲 · 一句话看懂整条线

LLM 强在「理解和生成」,弱在「可靠、实时、能动手、有记性、可控」。
整部演进史,就是不断地把 LLM 力所不及的部分用外部结构补上,同时把 LLM 关在它擅长的位置。

换句话说:Agent 本质上是一个「概率系统」嫁接在「确定性工程」上的产物 —— 模型负责猜,工程负责兜底。这就是后面每一个模块存在的根本原因。

01

先看大局:九个阶段,其实在解决三个递进的问题

第一层 · 知道什么
知识
让 LLM 突破「只知道训练数据」的局限 —— 记得住对话,也查得到实时与私有信息。
② 记忆③ RAG
第二层 · 能做什么
能力
让 LLM 从「只会说」变成「能动手」—— 会调工具,且工具多了也能精准、高效地用。
④ Function Call & MCP⑥ Skill

质量
让 Agent 从「能跑」变成「能用」—— 会编排多步任务、能协作、还能在真实世界稳定不崩。

02

九阶段演进时间轴

★ 枢纽 · 黄=怎么理解 · 绿=PM 拍板 · 📁=点开看案例
读法提示:每张卡底部都有一条可点开的 📁 贯穿例 —— 我们用同一个工单「接入方回调签名验证失败」贯穿全部九个阶段。想读原理就看卡片正文,想跟故事就逐个点开 📁,看这一单如何被一步步办结。完整连续版见文末「一页回看」,或独立文档《贯穿案例》。
1
LLM万能百科全书地基
两大致命缺陷
无记性 + 知识静态
上一句说"我叫张三",下轮就忘;且只知道训练截止前的世界,不懂你公司的文档
本质
海量文本的接龙预测
像一部博学但被锁住的百科全书,问知识性问题答得不错
为什么不够用
补不了这两条
「接个聊天框」解决不了 —— 这两个缺陷,直接催生了记忆和 RAG
一句话: 这是起点,也是所有痛点的源头。后面每个阶段,都在补它的某一条短板。
PM 在这拍的板: 先判断「这件事到底要不要上 Agent」。如果任务规则固定、不需要多步推理,传统软件 + 一次 LLM 调用就够了,别为了用 Agent 而用。Agent 的成本和不确定性,只有在"任务多步、依赖判断、流程会变"时才划算。
贯穿例 ①
接入方群里发来「回调签名验证一直失败,帮忙看下?」。把这句直接丢给裸 LLM,它只会泛泛地讲套话:「签名失败通常是密钥不对、参数排序错、时间戳过期……建议逐一排查。」话没错,但它不懂我们的签名算法、不知道这个接入方是谁、日志在哪 —— 一份网上随处可见的通用答案。
补「没记性」→ 知识层
2
记忆让对话连续知识层
驱动力
每次调用都"失忆"
LLM 每次 API 调用无状态,不带上历史就完全不认识你
新设计
短期 + 长期 + 进化
短期:上下文窗口管理(摘要/分层/虚拟分页);长期:跨会话持久化;高级:记忆→用户画像→生成技能
为什么这么设计
核心是"少让用户重复"
价值不在"能存多少",而在"减少重复表达";分类清晰 + 安全前置 + 演进有梯度
给 PM 的类比: 记忆是被动的(遇到才回忆),技能是主动的(条件反射式触发)。从「记住事实」→「总结规则」→「形成技能」是三级进化,正是新员工成长为熟练工的过程。对应你手册 Memory #10
PM 在这拍的板: 定「记什么、记多久、谁能看」。哪些信息值得跨会话记住(偏好、历史结论)、哪些用完即弃;长期记忆涉及用户数据,留存周期和隐私边界要先和合规对齐,别等上线才补。
贯穿例 ②
技术支持追问后补了一句「他们用的是 V2 签名,按这个继续」。有了记忆,Agent 记住了"这一单是 V2",整段对话不必反复重申,也记得这个接入方上周刚问过回调地址。但 V2 的算法细节它还是不知道 —— 那要靠下一步。
补「不知私事 + 会编」
3
RAG引入业务知识 / 最新信息知识层
驱动力
私有知识塞不进
记忆解决了连续性,但 LLM 仍只懂训练数据,不懂你的业务
新设计
先检索,再生成
从知识库检索相关片段拼进 prompt;进阶版(Self/Corrective RAG)先判断"证据够不够"
为什么这么设计
让回答有依据、可追溯
演进:朴素→优化检索→Agent驱动;不够就改写/兜底/承认"没找到依据"
给 PM 的类比: 普通 RAG 是"开卷考试照抄";Self-RAG 是"先判断这页资料到底能不能答这道题"。你手册"无依据就明说"正是后者。对应你手册 知识库 #7
PM 在这拍的板: 守住「没依据时宁可说不知道,也不许编」这条产品红线。同时盯知识库的质量与更新机制——文档不全或过期,再好的检索也救不回来。RAG 的天花板是知识库,不是模型。
贯穿例 ③
Agent 检索内部《签名规则文档》,捞到 V2 的真实步骤(参数字典序拼接、HMAC-SHA256、时间戳 5 分钟有效),据此回答:「V2 要求参数按字典序拼接后再签名,请先确认拼接顺序;另外时间戳超 5 分钟会失效。」若文档里没有,它会明说"未找到依据"而不是编。但它还没法真去看这一单的失败日志。
补「不会做事」→ 能力层
4
Function Call & MCP从「说」到「做」能力层
驱动力
只会说,没有手脚
RAG 让它能答"该查天气",但它没法真的去调天气 API
新设计
输出调用指令 + 标准协议
Function Calling:LLM 输出结构化调用指令,应用侧执行;MCP:把工具做成即插即用的标准协议
为什么这么设计
★ 决策与执行分离
LLM 只负责"决定调什么",代码负责"真执行" —— 这正是安全可控的根,也是 deterministic-picker 的起源
给 PM 的类比: 像点菜 —— 你(LLM)只负责下单(说要哪道菜、几份),后厨(代码)负责做菜。绝不让你自己冲进后厨。这道"防火墙"就是企业敢用 Agent 的前提。对应你手册 Tool Governance #8
PM 在这拍的板: 梳理「Agent 能调哪些工具、哪些算高危」。把每个工具按风险分级(只读 / 可写 / 不可逆),明确哪些要先确认再执行。这份工具清单是后面安全评审的源头,该 PM 主导,不能甩给开发随手定。
贯穿例 ④
Agent 这次不只是说,它输出一个调用指令:调日志查询接口、传入这个接入方的 appKey 和时间段;代码侧真去执行、把日志捞回来。它发现:「日志显示你们提交的 timestamp 比服务器晚了 8 分钟 —— 超过 5 分钟有效期。」注意:决定"查日志"的是 LLM,真正执行的是代码 —— 这道分离是后面所有安全的根。
补「谁来编排多步任务」→ 质量层
5
Agent流程编排:ReAct + 可选 Plan质量层
驱动力
一句话要分好几步
"把钉钉文档搬到语雀"需要先复制再粘贴 —— 谁来把多步任务串起来?
新设计
一个循环 loop
思考→执行→观察→再决定,直到完成。生产 90% 用「ReAct 主循环 + 按需 Plan」
为什么这么设计
简单零开销,复杂才规划
简单任务直接做(不浪费),复杂任务才先列计划 —— 灵活又可控
给 PM 的类比: 这就是 Agent 的"大脑"。你手册说的"不是越自主越好",就体现在这:能不规划就不规划,把自由发挥的空间压到最小。对应你手册 主链路
PM 在这拍的板: 划「哪些步骤交给 LLM 自由发挥,哪些必须写死成固定流程」。企业里已知稳定的流程应尽量固化,只在真需要判断的节点放开。自主度是产品决策,不是技术细节——放太开会跑偏,管太死又僵化。
贯穿例 ⑤
现在它能自己跑一个排查循环:查日志 → 看到时间戳异常 → 决定再核对参数拼接 → 调接口比对 → 发现拼接也有问题 → 汇总。查了第一步才知道第二步查什么,这正是 ReAct。结论:「两个病因:① 时间戳超时(主因);② 参数 amount 没参与签名。修复:校准时间 + 把 amount 纳入拼接。」
补「工具太多,选不准」→ 能力层
6
Skill精准执行任务能力层
驱动力
工具一多就崩
50 个工具的说明书占 5 万 token,既挤占上下文,又让 LLM 选择困难
新设计
渐进式加载
平时只看"有哪些能力"的摘要(~100 token/个),命中后才展开完整手册和工具(Advertise→Load→Read→Run)
为什么这么设计
10 倍 token 效率
大多数对话只用 1-2 个 skill;Skill = SOP + 工具 + 资源,打包成"能力包"
给 PM 的类比: 不是把整个工具箱倒在 LLM 面前,而是给新员工一份对应岗位的 SOP 手册 —— 它不是在"猜"怎么干,而是"按手册操作"。对应你手册 SkillLoader #6
PM 在这拍的板: 拆「业务该切成哪几个 Skill、每个 Skill 的 SOP 怎么写」。这步最吃业务理解,几乎是纯产品活——Skill 边界划得好不好,直接决定 Agent 准不准。SOP 是你把业务经验喂给 Agent 的接口。
贯穿例 ⑥
我们把"签名失败排查"做成一个 Skill(SOP + 专用工具 + 历史案例库)。Agent 一识别是签名问题就加载它,按固定步骤排查而非临场发挥。平时只看一句"我会排查签名问题"的摘要,命中才展开完整手册 —— 省下大量上下文。老师傅的经验,就这样固化成了可复用打法。
补「单个脑子带不动」→ 质量层
7
Multi-Agent专家协作,并行探索质量层
驱动力
单 Agent 三大困境
上下文膨胀(越聊越忘)、注意力分散(易跑偏)、串行瓶颈(任务排队慢)
新设计
分工 + 协作
七种模式:主从委托/接力/对等讨论/层级/竞争投票/评估反馈/动态路由,生产中常组合使用
为什么这么设计
不是越多越好
收益:上下文隔离+并行+专家化+交叉验证;但成本翻 N 倍 → 先单 Agent + 多 Skill,扛不住再拆
给 PM 的类比: 像把一个全能员工拆成一个项目组。质量更高,但开会、沟通、人力成本都涨。能一个人干完的别开会 —— 正是你手册的克制。对应你手册 Skill Router(克制版)
PM 在这拍的板: 忍住「不到必要不拆多 Agent」。默认单 Agent + 多 Skill,只有当单个真扛不住(上下文爆、角色互相干扰)才拆。拆一次,成本和复杂度翻 N 倍——这是性价比决策,不是越先进越好。
贯穿例 ⑦
假设这单升级:不只排查,还要同时分析该接入方近 7 天失败趋势、再写一份对外回复。一个脑子忙不过来,可拆成排查/分析/写作三专家 + 一个协调者。但只有真扛不住才拆 —— 单这一单,其实一个 Agent + 多 Skill 就够,不必开"项目组"。
补「上线就崩」→ 质量层
8
Harness让 Agent 稳定运行质量层
驱动力
没保护就跑不过 24 小时
LLM 错误率不低(工具格式错误 5-10%,幻觉更高);Demo 能跑,生产环境一遇意外就崩
新设计
包在外层的运行时护甲
六大子系统:错误分类与恢复 / 上下文工程 / 迭代控制 / 工具治理 / 安全防护 / 可观测性
为什么这么设计
★ 从"能跑"到"能用"的分水岭
它不改变 Agent 的决策,只负责让它"活得久、跑得稳" —— 企业落地的安全底座
给 PM 的类比: 如果 Agent Loop 是"大脑",Harness 就是免疫系统 + 骨骼 + 皮肤。你手册的 Audit、工具治理、人工审核、容错重试,合起来就是一套企业版 Harness。对应你手册 Audit #11 / Tool Governance / 人工审核 #12
PM 在这拍的板: 卡住「上线红线:可观测 + 高危人工审核 + 工具权限」三件套——缺一不可发布。这是你能不能签发布单的底线,六个子系统怎么逐条验收,见配套的《Harness 安全底座》专题图。
贯穿例 ⑧
真实流量里什么都可能出:日志接口偶发超时(自动重试)、模型反复调同一个挂掉的接口(迭代上限掐断)、它想调"重置接入方密钥"的高危操作(挡下,转人工审核)、整单排查轨迹全程留痕便于复盘(且 appsecret 不落日志)。护甲不改变它怎么排查,只保证它活得久、不闯祸、出事查得到。
形态演进:从服务端走到你的电脑里
9
Claw每个人的私人秘书(本地端)形态演进
驱动力
"告诉你怎么做"还得自己动手
传统 Agent 跑在服务端,只能给步骤,用户还得手动复制粘贴去执行
新设计
Agent 跑在你的电脑上
以本地客户端运行,直接读你的文件、跑命令、改代码、提 PR
为什么这么设计
闭合"决策-执行-验证"
消除"说"和"做"之间的断层 —— 它直接动手,并能验证结果(本质同服务端,核心是跑在了本地)
一句话: 这不是新原理,而是部署形态的演进 —— 从"远程顾问"变成"贴身秘书"。能力更强,但也意味着权限、安全的要求更高。
PM 在这拍的板: 评估「要不要把 Agent 放到用户本地、放了以后权限怎么收」。本地端能直接动文件、跑命令,体验强一截,但权限边界和用户授权要做得更重——能力越靠近用户的真实系统,出事的代价越大。
贯穿例 ⑨ · 收束
最终形态:Agent 跑在对接同学的电脑上,直接打开他们的代码、定位拼接签名那段、改好、跑一遍验证、提个 PR,而不是只甩一句"请把 amount 纳入签名"。"说"和"做"之间的断层被消除了 —— 但它能直接碰本地系统,所以权限和授权要求也最高。这一单,至此才算真正办结。

03

暗线:控制权的钟摆

控制权在「规则」与「LLM」之间摆动,最终收敛

早期 · 全用规则死板、不灵活
→
自主 Agent · 全交 LLM灵活,但失控、会崩
→
工程实践 · 收回代码/流程可控、可复盘、稳定
最终稳定在「LLM 负责判断、代码负责决策与兜底」—— 这就是 deterministic-picker,是"概率系统嫁接在确定性工程上"的具体落地,也是你手册从头讲到尾的那件事。
04

你真正要带走的

为什么这么设计 · 总答案

九个阶段,归到三层:知道什么(知识)、能做什么(能力)、怎么做得好(质量)。每个模块都是被"上一代的痛"逼出来的补丁 —— 没有一个是为炫技。

你手册的历史定位

你那份手册不是在描述「一种选择」,而是在复述一个被演进史验证过的收敛结论 —— 它是这条主线的「企业落地版」。

两条给 PM 的落地忠告(来自一线实践)

  • 观测比开发更重要。Agent 是概率系统,今天好好的链路,模型升级一版可能就悄悄退化了。先建好可观测体系(轨迹录制、成功率、badcase 归因),持续打磨 —— 这是常态,不是上线后的"维护"。
  • 对前沿架构保持敏感。这个领域半年一代际。一个好的架构思路(如渐进式加载、结构化压缩)往往比堆人力调参有效 10 倍。
