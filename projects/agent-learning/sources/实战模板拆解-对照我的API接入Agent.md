## 1. ⭐ rag_failure_diagnostics_clinic —— 你的「联调排查」骨架

**它在做什么**：你把一段真实 bug 描述贴进去，LLM 把它归类到一个**预定义的 12 种失败模式库（P01–P12）**里，并给出"最小结构化修复方案"。

**为什么这是你联调排查的现成模板**：把它的"RAG 失败模式库"换成你的"API 联调问题库"（签名失败 / 回调异常 / 续租失败…），逻辑一模一样。

**能直接抄的 3 个设计思路**：

1. **它就是 deterministic-picker 的活样本**（对应你 #3/#4）：
   System Prompt 里硬写死「你**只能**从 P01–P12 里选一个主分类，**不许发明新 id**」。
   > 抄点：你的需求识别也该这样——让 LLM 从你定义好的 `request_type` 枚举里选，而不是自由生成。这是"LLM 填表、不自己拍板"在 prompt 层的具体写法。

2. **"问题模式库"= 你的 Skill 案例库**（对应你 #6/#10 Memory）：
   它把每种失败模式写成 `{id, name, summary}` 的结构化清单喂给模型。
   > 抄点：你的"签名失败排查 Skill"也可以这样维护成一个结构化案例库，Memory 里沉淀的真实联调案例直接往里加。

3. **强制输出结构 + 落盘成 JSON 报告**（对应你 #11 Audit）：
   要求模型按"主分类 / 次选 / 推理 / 最小修复"四段输出，并存成 `rag_failure_report.json` 当增量案例库。
   > 抄点：你的排查输出也固定成结构化四段，技术支持能直接拿去用，同时天然形成可复盘记录。

**看哪里**：`rag_failure_diagnostics_clinic.py` 的 `PATTERNS` 列表 + `build_system_prompt()` 函数。**这是 5 个里最该先读的。**

## 2. rag_database_routing —— 你的「分诊 + 派单 + 检索」

**它在做什么**：上传的文档分别进**3 个专用库**（产品信息 / 客服FAQ / 财务）；用户提问时，一个 router agent 自动判断该查哪个库；**3 个库都没命中就兜底走联网搜索**。

**对应你的模块**：#2 Session Router + #6 Skill Router + #7 知识库检索，一次性串起来了。

**能直接抄的设计思路**：

- **"路由 + 多库 + 兜底"三件套**：你的 API 文档、错误码说明、续租规则、号池规则本来就是不同知识域，正好对应"多个专用库 + 路由"。
- **兜底机制**：它"没命中就联网"，你的版本应该改成"**没命中就明说'文档未找到依据'**"（你手册里的安全口径，比联网更稳）。

**看哪里**：`rag_database_routing.py` 里 router agent 的判断逻辑 + fallback 分支。

## 3. corrective_rag —— 你的「知识库检索」标准件（Self-RAG/CRAG）

**它在做什么**：检索文档 → 用 LLM **给每篇文档的相关性打分** → 不够相关就**改写查询重试** → 还不行就**联网兜底** → 最后生成答案。整个流程用 **LangGraph 状态机**编排。

**对应你的模块**：#7 知识库检索的"进阶版"。这正是附录里说的 **Self-RAG / CRAG** 范式的真实代码。

**能直接抄的设计思路**：

- **"检索完先判断证据够不够"这一步**：普通 RAG 检索完直接塞给模型生成；它多了一道"相关性评分"关卡。
  > 抄点：你回答 API 字段含义前，先让模型判断"检索到的片段是否真能回答"，不够就走"未找到依据"，直接落实你的可追溯口径。
- **LangGraph 编排**：和你手册"Task State 状态机"的技术地基是同一个东西，可以顺带看状态机长啥样。

**看哪里**：`corrective_rag.py` 里的 grading 节点和 LangGraph 图定义。

## 4. ai_data_analysis_agent —— 你的「单量分析」任务链

**它在做什么**：上传 CSV/Excel → 自动推断 schema → 用户用自然语言提问 → 模型转成 **SQL（DuckDB 执行）** → 出结果和可视化。

**对应你的任务链**：单量分析（确认口径 → 趋势分析 → 异常判断 → 输出周报）。

**能直接抄的设计思路**：

- **自然语言 → SQL → 真实计算**：你的"近 7 天单量"本质就是这个。关键是**让 DuckDB 算数、让 LLM 只负责翻译和解读**——又一次 deterministic-picker：算数交给确定性引擎，别让 LLM 心算。
- **schema 自动推断**：对应你"确认口径"——先让系统把字段/口径摆清楚，再分析。

**看哪里**：`ai_data_analyst.py` 里自然语言转 SQL 的 prompt + DuckDB 执行部分。

## 5. ⭐ trust_gated_agent_team —— 你的「Tool Governance + Audit + 人工审核」

**它在做什么**：一个多 agent 流水线（Researcher→Analyst→Writer），但**每个 agent 参与前必须通过信任打分（0–100，金/银/铜分级）**，不达标直接被挡；**每一步操作都写进 SHA-256 哈希链审计日志**，任何一条被篡改后面全部对不上。

**对应你的模块**：#8 Tool Governance（够格才能调）+ #11 Audit（防篡改可复盘）+ #12 人工审核（高风险闸门）。

**能直接抄的设计思路**：

- **"参与前先过闸"的 gating 模式**：你的"凭证发放/上线/改配置"前的人工审核，就是这种"达标才放行"的闸门，可以参考它怎么用代码把闸门做实。
- **哈希链审计**：你手册说 Audit 要"可复盘、可审计"，它给了一个**防篡改**的具体实现思路——对涉及凭证、上线的敏感操作特别有说服力。
  > 注意：你的版本要叠加"敏感信息不进日志"（appsecret 不落盘），这是它没强调、你更严的地方。

**看哪里**：README 里的架构图 + 源码里 trust check 的 gating 逻辑。

## 学习路线建议（按优先级）

1. **先读 #1（failure_diagnostics）** —— 最贴近你、代码最短、deterministic-picker 看得最清楚。
2. **再读 #3（corrective_rag）** —— 看 Self-RAG + LangGraph 状态机，补你 #7 和 Task State 的实现认知。
3. **#2 + #4 + #5 按需** —— 分别对应路由、单量分析、安全闸门。

**统一抄法**：每个模板都问自己三个问题 ——
① 它把"判断"交给了 LLM 还是代码？
② 它的兜底（没命中/不达标）怎么处理？
③ 它沉淀了什么可复盘的记录？
这三问正好对应你手册的 deterministic-picker、安全边界、Audit 三大支柱。
