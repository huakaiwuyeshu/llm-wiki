为什么是 Agent,为什么是现在

在讲"Agent 由哪几块组成"之前,先回答每个 PM 心里的三个问题:它和我们以前做的产品到底差在哪?为什么偏偏是现在?作为 PM,我的活儿会变成什么样?这三问答不好,后面再精彩也抓不住人。

放在《Agent 演进图》之前讲
一句话先抓住注意力

过去我们做的产品,是把流程写死、让用户照着点。
Agent 第一次让我们能做「给个目标,让它自己想办法办成」的产品。

这不是又一个技术名词,而是产品形态本身变了 —— 从"设计每一步操作",变成"设计一个会自己决策的员工"。下面三个问题,就是把这句话讲透。

Q1

它和"普通调了 AI 的产品"差在哪?

以前 · 调一次 API 的产品
你设计好每一步,AI 只是其中一环
  • 流程是你写死的:用户点 A→填 B→提交 C,路径固定
  • AI 只做一件小事:翻译这段、总结这篇、分个类
  • 不会用工具、没有记性:每次调用都是一锤子买卖
  • 出错是确定的:同样输入永远同样输出,好测
现在 · Agent 产品
你给目标,它自己拆步骤、调工具、办成
  • 流程是它自己决定的:同一个目标,它可能走不同路径
  • 会动手:能查库、调接口、改文件,不只是"说"
  • 有记性、能多步:查了 A 才知道下一步该查 B
  • 结果是概率的:这正是它强大的地方,也是要"上护甲"的原因
Q2

为什么偏偏是现在能做了?

①模型终于"听得懂指令、还会用工具"了

三年前的模型只会续写文本,你没法可靠地让它"输出一个调用天气接口的指令"。现在的模型能稳定地理解任务、输出结构化的工具调用 —— 这是 Agent 能"动手"的前提,也是 2023 年之后才真正成熟的能力。

②工程实践摸索出了"让它不失控"的办法

光有聪明的模型不够。早期大家一度把控制权全交给模型,结果是失控、会崩、没法上线。这两年沉淀出一套确定性工程(决策与执行分离、流程固化、运行时护甲),才让 Agent 从"demo 惊艳"走到"敢上生产"。

模型负责猜,工程负责兜底 —— 两件事同时成熟,才有了今天能落地的 Agent。

③所以它不是"会过去的风口",是新的产品底座

它解决的是一类以前根本没法用软件做的事:流程多变、依赖判断、需要边做边调整的任务。这类需求一直都在,只是过去只能靠人。能力底座变了,值得我们认真投入,而不是观望。

Q3

那作为 PM,我的活儿会变成什么样?

这是听众最关心的一题
以前:画流程图,定义每一步用户怎么点
→
现在:定义目标和边界 —— 它能做什么、绝不能做什么
以前:写清楚每个按钮的交互逻辑
→
现在:写 SOP / 划工具权限 —— 把业务经验喂给 Agent
以前:功能测通了就能上
→
现在:盯成功率和 badcase —— 概率系统要持续观测、迭代
以前:需求评审时和开发对功能
→
现在:评审时拍安全红线 —— 高危动作要不要人工审核
承上启下 · 一句话过渡到主线

你的核心工作没消失,只是从"设计操作"上移到了"设计判断与边界"。
而 Agent 怎么一步步演化成今天这个样子、每一步又需要 PM 拍哪些板 —— 这正是接下来《Agent 演进图》要讲的九个阶段。
