内部学习稿 · 一个例子讲透九阶段

同一个案子,走完九个阶段

演进图讲的是抽象原理,这里换种讲法:拿一个真实的联调排查工单,让它从头穿过九个阶段。每一阶段只看一件事 —— 到这一步,Agent 多了什么能力,这个案子因此能往前走一格。同一个故事讲九遍,比九个名词好记。

配合《Agent 演进图》九阶段对照看
案情 · 贯穿全文的那一个工单

某接入方的技术对接群里发来一句:
「你们的回调签名验证一直失败,接口对接卡住了,帮忙看下?」

这是你手册「联调排查」链路最典型的一单。我们就看:从一个只会聊天的 LLM,到能真正把这单办结的 Agent,中间每一阶段补上了什么。

▸

这一单的九步成长

绿块 = 这阶段补了什么 · 红块 = 缺它会怎样
1
LLM只会聊天
把这句话直接丢给一个裸 LLM,它会泛泛地讲: 「签名失败通常有几种原因:密钥不对、参数排序错误、时间戳过期……建议逐一排查。」 话没错,但它不知道我们的签名算法是什么、这个接入方是谁、日志在哪 —— 等于一份网上随处可见的通用答案。
这阶段的能力
理解人话、生成通顺回答
这一单卡在哪
不懂我们的业务、不能查、没记性,只能给套话
想让它懂我们的签名规则 →
2
记忆记得住对话
技术支持追问:「他们用的是 V2 签名,你按这个继续。」有了记忆,Agent 记住了"这一单是 V2 签名",后面整段对话不用反复重申;也记得这个接入方上周刚问过回调地址的事。
这阶段的能力
跨轮次、跨会话记住上下文,不再每句话都失忆
仍然卡在哪
记住了"是 V2",但V2 的算法细节它还是不知道
V2 签名规则写在内部文档里 →
3
RAG查得到内部文档
Agent 去检索内部《签名规则文档》,捞到 V2 的具体步骤(参数字典序拼接、HMAC-SHA256、时间戳 5 分钟有效),据此回答: 「V2 要求参数按字典序拼接后再签名,请先确认你们的拼接顺序。另外时间戳超过 5 分钟会失效。」 如果文档里没有,它会明说"未找到依据",而不是编一个。
这阶段的能力
回答基于我们的真实文档,有据可查、不瞎编
仍然卡在哪
能讲规则,但没法真的去看这单的失败日志
得让它真的去查这一单的日志 →
4
Function Call & MCP能动手调接口
Agent 这次不只是"说",它输出一个调用指令:调日志查询接口、传入这个接入方的 appKey 和时间段。代码侧真的去执行、把日志捞回来。它发现: 「日志显示:你们提交的 timestamp 比服务器时间晚了 8 分钟 —— 超过了 5 分钟有效期。」 注意:决定"查日志"的是 LLM,真正执行查询的是代码 —— 这道分离是后面所有安全的根。
这阶段的能力
从"只会说"到"能调真实接口拿真实数据"
仍然卡在哪
单次调一个接口够了,但这一单要查日志→对参数→比时间,好几步
多步排查,谁来串起来 →
5
Agent Loop自己串多步
现在它能自己跑一个排查循环:查日志 → 看到时间戳异常 → 决定再核对一下参数拼接顺序 → 调接口比对 → 确认拼接也有个小问题 → 汇总。查了第一步才知道第二步查什么 —— 这正是 ReAct「想→做→看→再想」。
「定位到两个问题:① 时间戳超时(主因);② 参数 amount 没参与签名。修复:校准服务器时间 + 把 amount 纳入拼接。」
这阶段的能力
把多步排查自动串成一条链,边查边调整方向
仍然卡在哪
这单它会查了,但"续租失败""号池耗尽"等几十种问题,它每次都从零摸索
把"签名失败排查"沉淀成固定打法 →
6
Skill按 SOP 精准排查
我们把"签名失败排查"做成一个 Skill(SOP + 专用工具 + 历史案例库)。Agent 一识别出是签名问题,就加载这个能力包,按固定步骤排查,而不是临场发挥。平时它只看一句"我会排查签名问题"的摘要,命中了才展开完整手册 —— 省下大量上下文。
这阶段的能力
把老师傅的排查经验固化成可复用的 SOP,又快又稳又省 token
仍然卡在哪
单个 Agent 同时背排查、分析、写报告多个角色时会精神分裂
复杂工单要多个专家协作 →
7
Multi-Agent专家分工(按需)
假设这一单升级了:不只是排查,还要同时分析这个接入方近 7 天的失败趋势、再写一份对外回复。一个脑子忙不过来,于是拆成:排查专家 + 数据分析专家 + 写作专家,一个协调者统筹。但只有真扛不住才拆 —— 单这一单,其实一个 Agent + 多 Skill 就够,不必开"项目组"。
这阶段的能力
复杂任务可拆给多个专家并行,各管一摊、互不干扰
要警惕
拆一次成本和协调复杂度翻倍,不到必要别拆
这些能力要真上线,扛得住意外吗 →
8
Harness上线护甲
真实流量里什么都可能出:日志接口偶发超时(自动重试)、模型想反复调同一个挂掉的接口(迭代上限掐断)、它想调一个"重置接入方密钥"的高危操作(挡下来,转人工审核)、整单的排查轨迹全程留痕便于复盘(且 appsecret 不落日志)。这层护甲不改变它怎么排查,只保证它活得久、不闯祸、出事查得到。
这阶段的能力
从"demo 能跑"到"敢上生产" —— 容错、限流、审核、可观测
形态还能再进一步
它仍在服务端,给出修复建议后还得人去操作
让它直接动手,而不只是给建议 →
9
Claw本地端 · 直接动手
最终形态:Agent 跑在对接同学的电脑上,直接打开他们的代码、定位拼接签名那段、改好、跑一遍验证、提个 PR,而不是只甩一句"请把 amount 纳入签名"。"说"和"做"之间的断层被消除了 —— 但它能直接碰本地系统,所以权限和授权要求也最高。
这阶段的能力
闭合决策→执行→验证,从"远程顾问"变"贴身秘书"
代价
能力越贴近真实系统,权限边界和安全要求越重

一页回看:每一步给这单加了什么

1LLM给通用套话,不懂我们业务
2记忆记住"这单是 V2 签名",对话连续
3RAG查内部文档,讲出 V2 的真实规则,没依据就明说
4Function Call真的去调日志接口,发现时间戳超时
5Agent Loop自己串多步:查日志→比参数→定位双重病因
6Skill把排查打法固化成 SOP,又快又稳又省
7Multi-Agent复杂时拆排查/分析/写作专家(按需,克制)
8Harness容错重试、掐死循环、高危转人工、全程留痕
9Claw本地直接改代码、验证、提 PR,闭合执行
