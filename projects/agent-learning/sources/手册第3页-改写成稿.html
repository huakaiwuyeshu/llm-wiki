Agent 是怎么演进到业务 Agent 的

先破一个最常见的误解:一个固定流程,不会因为中间接了大模型,就自动变成智能体;会调用工具,也不等于就是 Agent。Agent 不是一个固定形态的产品,而是它从早期的工具调用,一步步长出记忆、检索、动手、编排、协作、护甲等能力,最终成为可以沉淀经验的业务系统。
读法很简单:跟着下面这条主线走一遍 —— 看每一步,是被上一代的什么短板逼出来的。走完,你自然会看懂它为什么落到「混合架构」。

01

主线:九步演进,每一步都在补上一代的短板

★ 枢纽 · 从上往下读 · 右侧 = 对应手册模块
1LLM万能百科起点:能理解和生成,但无记性、知识静态、不会动手 —— 后面每一步都在补它的某条短板。地基
2记忆Memory补「每次调用都失忆」—— 短期上下文 + 长期持久化,让对话和经验连续。知识层手册 #10
3RAG检索增强补「不懂私有业务、还会编」—— 先检索再回答,没依据就明说未找到。知识层手册 #7
4Function Call& MCP★ 补「只会说不会做」—— LLM 输出调用指令、代码负责执行。决策与执行分离,是安全可控的根。能力层手册 #8
5Agent LoopReAct★ 补「多步任务谁来串」—— 想→做→看→再决定的循环。Agent 不是一种产品形态,而是一种控制策略 —— 就是这个循环本身。质量层主链路
6Skill能力包补「工具一多就选不准」—— 渐进式加载 SOP + 工具,按手册操作而非临场猜。能力层手册 #6
7Multi-Agent多智能体补「单个脑子带不动」—— 拆成专家协作;但成本翻倍,不到必要不拆。质量层手册 #6
8Harness运行时护甲★ 补「上线就崩」—— 容错、限流、工具治理、人工审核、可观测。从「能跑」到「能用」的分水岭。质量层#8/#11/#12
9Claw本地端部署形态演进 —— Agent 跑到你电脑上,直接动手并验证,闭合决策-执行-验证。形态

换个粗看法:按「自主度」分四档,帮你选型

手册原本的演进图 · 上面九步是"为什么",这四档是"该用到哪一档"
第 1 档
早期 Agent
聊天 + 工具调用

第 2 档
工作流 Agent

同样是「记忆」「Skill」,在九步里靠前(按能力补齐的先后排),在四档里却靠后(按自主度高低排)。轴不一样,位置自然会变 —— 别纠结顺序,看它解决什么问题。

04

真正决定架构的暗线:控制权的钟摆

★ 这才是"为什么混合架构"的根
早期 · 全用规则流程写死,死板、不灵活,稍微变一点就得改代码。
→
自主 Agent · 全交 LLM把控制权全给模型,灵活了,但失控、会跑偏、上线就崩。业务最怕的不是系统不够聪明,而是系统自作聪明。
→
工程实践 · 收回控制权把确定的部分用代码/流程固化,只在需要判断处放 LLM。可控、可复盘。
钟摆最终收敛在 「Workflow 负责边界,Agent 负责弹性」 —— Workflow 规定哪些步骤必须走、哪些权限不能越、哪些结果必须审核;Agent 在允许的范围内处理不确定性。这就是混合架构。
所以它不是某种偏好,而是整个行业试错之后的收敛终点:能力可以一直长(前面九步),但控制权必须留在确定性工程手里。
佐证 · 印证上面的结论

为什么我们也选混合架构

钟摆讲的是行业为什么都走向混合架构(通用规律)。再看一眼我们自己为什么也该选它:API 接入、联调排查、单量分析,不是开放闲聊,而是高频、确定、可审核的流程。它的每一条业务特征,恰好都对应混合架构里的一块 —— 这正是"把模型放进可控流程"在我们这儿的具体长相。

业务特征因此 Agent 需要对应模块（后文详解）
流程顺序稳定不要每次重想流程Workflow
字段规则明确判断可校验Schema + Validator
问题高度重复复用专家经验Skill + 知识库
任务多轮推进知道走到哪一步Task State
凭证敏感、上线敏感必须人工确认Audit + 人工审核

承上启下:右列这几个模块,就是手册后面要逐个拆开讲的东西 —— 演进史告诉你它们为什么存在,接下来的章节告诉你它们具体怎么做。

讲给业务同学的一句话:Agent 不是越自主越好。对企业业务来说,更重要的是把大模型放进可控流程里 —— LLM 负责理解和表达,Workflow / Schema / Skill / Task State / Memory / Audit 和人工审核负责稳定、可控、可复盘。
