内部学习稿 · Harness 专题拆解

Harness · 企业落地的安全底座

演进图第 ⑧ 阶段的放大版。Harness 不改变 Agent 怎么决策,只负责让它「活得久、跑得稳、不闯祸」。它由六个子系统组成,每一个都在挡一类具体的线上风险 —— 这张图,就是评审时该逐条拍板的清单。

配合《Agent 演进图》第 ⑧ 阶段使用
为什么单独画这张

Demo 能跑,不等于能上线。LLM 工具调用格式错误率有 5–10%,幻觉更高 —— 没有护甲,Agent 撑不过 24 小时真实流量。

Harness 就是包在 Agent 决策核心外面的那层「运行时护甲」。对 PM 来说,它对应的不是某个炫酷功能,而是你手册里最该补齐的那块:Audit、工具治理、人工审核、容错重试 —— 合起来就是一套企业版 Harness。

01

它在架构里的位置:包在决策核心的外面

HARNESS · 运行时护甲
Agent Loop（决策核心）思考 → 调用工具 → 观察 → 再决定

关键认知:Harness 不参与「调哪个工具」这种决策 —— 那是大脑的事。它只在决策的外圈做拦截、校验、重试、记录。大脑可以是概率的、会犯错的;护甲必须是确定的、不出错的。这正是「概率系统嫁接在确定性工程上」最直接的体现。

02

六个子系统,按它们回答的三个问题归类

颜色 = 所属问题
问题一
怎么不崩?
可靠性 · 出错也能自己缓过来
问题二
怎么不闯祸?
安全性 · 危险动作有闸门
问题三
怎么看得见?
可控性 · 出了事查得到、复得了盘
1
错误分类与恢复
Error Handling & Recovery
不崩
挡的风险
工具返回格式错、超时、接口偶发 5xx —— 任何一个未处理的异常都会让整条任务当场中断。
具体做什么
  • 把错误分级:可重试(超时)/ 可纠正(格式错)/ 必须中止(权限拒绝)
  • 可重试的自动退避重试,格式错的回喂让模型自己改
  • 真不行就优雅降级,而不是抛栈崩掉
对应你手册 · 容错重试机制
2
迭代控制
Iteration / Loop Control
不崩
挡的风险
Agent 陷入死循环:反复调同一个失败的工具,既烧 token 又永远不结束 —— 一次失控可能就是一笔可观的账单。
具体做什么
  • 设最大迭代步数 / 超时 / token 预算上限,到顶强制收尾
  • 检测重复动作(同样的调用连续出现就打断)
  • 触顶时给出「已尽力」的兜底答复,而非空转
对应你手册 · 主链路的步数/预算约束
3
上下文工程
Context Engineering
不崩
挡的风险
对话越跑越长,上下文超窗 → 要么报错,要么把关键信息挤掉导致「越聊越笨」、答非所问。
具体做什么
  • 历史摘要、分层、虚拟分页,把窗口管在安全水位
  • 每步只喂「这一步需要的」信息,降噪
  • 结构化压缩,而非简单截断丢尾巴
对应你手册 · 记忆管理 #10(短期上下文)
4
工具治理
Tool Governance
不闯祸
挡的风险
模型「决定」调用一个高危工具(删库、扣款、改配置),如果照单执行,一次幻觉就是一次生产事故。
具体做什么
  • 工具分级 + 权限校验,模型只能调被授权的
  • 高危操作先 dry-run / 二次确认,再真正执行
  • 参数校验把关,挡住明显越界的调用
对应你手册 · Tool Governance #8(受控工具)
5
安全防护
Safety & Human Gate
不闯祸
挡的风险
提示注入、越权数据访问、敏感信息外泄;以及「不可逆动作」被自动执行后无法挽回。
具体做什么
  • 输入/输出过滤,识别注入与敏感数据
  • 高风险、不可逆动作设人工审核闸门(Human-in-the-Loop)
  • 权限沙箱:限定 Agent 能碰的数据与系统边界
对应你手册 · 人工审核发布 #12
6
可观测性
Observability
看得见
挡的风险
线上出了 badcase,却不知道它哪一步想错了、为什么这么干 —— 无法复盘,就无法改进,问题只会反复出现。
具体做什么
  • 全链路轨迹录制:每步的思考、调用、结果都留痕
  • 成功率、耗时、token 等指标监控
  • badcase 可回放、可归因,且不落敏感信息
对应你手册 · Runtime Logs / Audit #11
03

拿去评审的检查清单:这六问没答全,就别说"能上线"

工具调用失败时,系统会重试 / 纠正 / 降级,而不是直接崩?#1 错误恢复问开发:格式错误、超时、5xx 分别怎么处理?
有没有最大步数 / token 预算上限,防止死循环烧钱?#2 迭代控制问开发:一个任务最坏情况花多少 token?触顶怎么收场?
长对话会不会超窗变笨?上下文怎么管?#3 上下文工程问开发:聊到第 50 轮还准不准?靠摘要还是截断?
高危工具(删除/扣款/改配置)有没有权限分级和执行前关卡?#4 工具治理问开发:模型能直接调的工具清单是什么?哪些要 dry-run?
不可逆动作有没有人工审核闸门?能挡住提示注入吗?#5 安全防护问开发:哪些动作必须人点一下确认?敏感数据怎么隔离?
线上 badcase 能不能回放、归因?日志会不会存敏感信息?#6 可观测性问开发:出了问题多久能定位到是哪一步想错的?
04

不用一次做全:哪些是上线前提,哪些可以迭代

上线前必须有(红线)

#4 工具治理 + #5 安全防护 + #6 可观测性。这三块管的是「不闯祸」和「出事查得到」—— 缺了它们,一次幻觉就可能是一次真实事故,且事后无从追责。这是企业敢不敢按下发布键的底线。

可以随流量迭代(体验线)

#1 错误恢复 + #2 迭代控制 + #3 上下文工程。这三块管的是「稳和省」—— 直接影响成功率、成本和长对话体验。初版可以先做基础版,再根据真实 badcase 持续打磨。观测先行,数据驱动迭代正是这里。
