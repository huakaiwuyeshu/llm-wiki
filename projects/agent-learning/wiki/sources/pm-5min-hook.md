## 核心内容

### 一句话钩子(Thesis)
过去做的产品是**把流程写死、让用户照着点**;Agent 第一次让我们能做**「给个目标,让它自己想办法办成」**的产品。这不是又一个技术名词,而是产品形态本身变了——从"设计每一步操作"变成"设计一个会自己决策的员工"。

### Q1 它和"普通调了 AI 的产品"差在哪?
| 以前·调一次 API 的产品 | 现在·Agent 产品 |
|---|---|
| 流程是你写死的:点 A→填 B→提交 C,路径固定 | 流程是它自己决定的:同一目标可能走不同路径 |
| AI 只做一件小事:翻译/总结/分类 | 会动手:查库、调接口、改文件,不只是"说" |
| 不会用工具、没有记性:每次调用一锤子买卖 | 有记性、能多步:查了 A 才知道下一步该查 B |
| 出错是确定的:同输入同输出,好测 | 结果是概率的:这正是强大之处,也是要"上护甲"的原因 |

### Q2 为什么偏偏是现在能做了?
1. **模型终于"听得懂指令、还会用工具"了**:三年前模型只会续写文本,无法可靠输出"调用天气接口的指令";现在能稳定理解任务、输出结构化工具调用——2023 年之后才真正成熟的能力
2. **工程实践摸索出"让它不失控"的办法**:早期把控制权全交模型→失控、会崩、没法上线;这两年沉淀出确定性工程(决策与执行分离、流程固化、运行时护甲),才从"demo 惊艳"走到"敢上生产"。金句:**模型负责猜,工程负责兜底——两件事同时成熟,才有了今天能落地的 Agent**
3. **不是"会过去的风口",是新的产品底座**:它解决一类以前根本没法用软件做的事——流程多变、依赖判断、需要边做边调整的任务;这类需求一直在,过去只能靠人

### Q3 PM 的活儿变成什么样?(听众最关心)
| 以前 | 现在 |
|---|---|
| 画流程图,定义每一步用户怎么点 | **定义目标和边界**——它能做什么、绝不能做什么 |
| 写清楚每个按钮的交互逻辑 | **写 SOP / 划工具权限**——把业务经验喂给 Agent |
| 功能测通了就能上 | **盯成功率和 badcase**——概率系统要持续观测、迭代 |
| 需求评审时和开发对功能 | **评审时拍安全红线**——高危动作要不要人工审核 |

### 承上启下(收尾)
PM 的核心工作没消失,只是**从"设计操作"上移到了"设计判断与边界"**。Agent 怎么一步步演化、每步 PM 拍哪些板——交给接下来的《Agent 演进图》九阶段。

## 关键细节
- 三问结构:Q1 差在哪 / Q2 为什么是现在 / Q3 我的活儿
- 时间锚点:模型可靠的结构化工具调用是 2023 年之后成熟的;"三年前的模型只会续写文本"
- Q3 四行角色转变表与演进图九条"PM 拍板"一一呼应(目标边界↔阶段①⑤、SOP/权限↔⑥④、成功率/badcase↔⑧观测、安全红线↔⑧人工审核)
