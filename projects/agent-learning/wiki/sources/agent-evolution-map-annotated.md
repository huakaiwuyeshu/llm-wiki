## 核心内容

### 总纲(Thesis)
LLM 强在「理解和生成」,弱在「可靠、实时、能动手、有记性、可控」。整部演进史 = 不断把 LLM 力所不及的部分用外部结构补上,同时把 LLM 关在它擅长的位置。Agent 本质 = **「概率系统」嫁接在「确定性工程」上** —— 模型负责猜,工程负责兜底。

### 三层递进框架(01)
| 层 | 问题 | 名称 | 包含阶段 |
|---|---|---|---|
| 第一层 | 知道什么 | 知识 | ② 记忆、③ RAG |
| 第二层 | 能做什么 | 能力 | ④ Function Call & MCP、⑥ Skill |
| 第三层 | 怎么做得好 | 质量 | ⑤ Agent Loop、⑦ Multi-Agent、⑧ Harness |

(① LLM = 地基,⑨ Claw = 形态演进,不入三层)

### 九阶段时间轴(02)
每张卡三栏:驱动力(pain)/ 新设计(design)/ 为什么这么设计(why);加黄条「给 PM 的类比」、绿条「PM 在这拍的板」、📁 贯穿案例切片(工单:接入方回调签名验证失败)。★ 标记枢纽阶段 = ④⑤⑧。

**① LLM(地基,"万能百科全书")**
- 缺陷:无记性 + 知识静态(上一句说"我叫张三"下轮就忘;只知训练截止前的世界,不懂公司文档)
- 本质:海量文本的接龙预测,博学但被锁住的百科全书
- PM 拍板:先判断**「这件事到底要不要上 Agent」**——任务规则固定、不需多步推理时,传统软件 + 一次 LLM 调用就够;Agent 只在"任务多步、依赖判断、流程会变"时才划算
- 案例①:裸 LLM 对签名失败只能给通用套话(密钥/排序/时间戳逐一排查),不懂自家签名算法、不知接入方是谁、日志在哪

**② 记忆(知识层)**
- 驱动力:每次 API 调用无状态,不带历史就"失忆"
- 设计:短期(上下文窗口管理:摘要/分层/虚拟分页)+ 长期(跨会话持久化)+ 进化(记忆→用户画像→生成技能)
- Why:核心是"少让用户重复",价值不在能存多少
- PM 类比:记忆是**被动的**(遇到才回忆),技能是**主动的**(条件反射触发);「记住事实→总结规则→形成技能」三级进化 = 新员工成长为熟练工。映射:手册 Memory #10
- PM 拍板:定「记什么、记多久、谁能看」;长期记忆留存周期和隐私边界要**先和合规对齐**
- 案例②:记住"这一单是 V2 签名",也记得该接入方上周问过回调地址;但 V2 算法细节仍不知道

**③ RAG(知识层)**
- 驱动力:私有知识塞不进,LLM 不懂你的业务
- 设计:先检索再生成;进阶版(Self-RAG / Corrective RAG)先判断"证据够不够"
- Why:回答有依据、可追溯;演进:朴素→优化检索→Agent 驱动;不够就改写/兜底/承认"没找到依据"
- PM 类比:普通 RAG = 开卷考试照抄;Self-RAG = 先判断这页资料能不能答这题。映射:手册 知识库 #7
- PM 拍板:守住「没依据宁可说不知道,不许编」红线;盯知识库质量与更新机制——**RAG 的天花板是知识库,不是模型**
- 案例③:检索内部《签名规则文档》得 V2 真实步骤(参数字典序拼接、HMAC-SHA256、时间戳 5 分钟有效)

**④ Function Call & MCP(能力层,★ 枢纽)**
- 驱动力:只会说,没有手脚(能答"该查天气",不能真调天气 API)
- 设计:Function Calling = LLM 输出结构化调用指令、应用侧执行;MCP = 工具的即插即用标准协议
- Why(★):**决策与执行分离** —— LLM 只决定调什么,代码真执行;安全可控的根,deterministic-picker 的起源
- PM 类比:点菜——LLM 只下单,后厨(代码)做菜,绝不让它冲进后厨;这道防火墙是企业敢用 Agent 的前提。映射:手册 Tool Governance #8
- PM 拍板:梳理工具清单并按风险分级(只读/可写/不可逆),明确哪些先确认再执行;**这份清单是安全评审的源头,该 PM 主导**
- 案例④:调日志接口发现 timestamp 比服务器晚 8 分钟(超 5 分钟有效期)

**⑤ Agent Loop(质量层,★ 枢纽;ReAct + 可选 Plan)**
- 驱动力:一句话任务要分好几步("钉钉文档搬到语雀"),谁来串
- 设计:思考→执行→观察→再决定的循环;**生产 90% 用「ReAct 主循环 + 按需 Plan」**
- Why:简单任务零开销直接做,复杂才规划——灵活又可控
- PM 类比:Agent 的"大脑";"不是越自主越好",能不规划就不规划。映射:手册 主链路
- PM 拍板:划「哪些步骤 LLM 自由发挥、哪些写死成固定流程」;**自主度是产品决策,不是技术细节**
- 案例⑤:自跑排查循环,定位双重病因:① 时间戳超时(主因)② 参数 amount 没参与签名;修复 = 校准时间 + amount 纳入拼接

**⑥ Skill(能力层)**
- 驱动力:工具一多就崩——50 个工具说明书占 5 万 token,挤上下文 + 选择困难
- 设计:渐进式加载(Advertise→Load→Read→Run),平时只看摘要(~100 token/个),命中才展开完整手册和工具
- Why:**10 倍 token 效率**;大多数对话只用 1-2 个 skill;Skill = SOP + 工具 + 资源的"能力包"
- PM 类比:给新员工对应岗位的 SOP 手册,"按手册操作"而非"猜"。映射:手册 SkillLoader #6
- PM 拍板:业务切成哪几个 Skill、SOP 怎么写——最吃业务理解,几乎纯产品活;**SOP 是 PM 把业务经验喂给 Agent 的接口**
- 案例⑥:"签名失败排查"做成 Skill(SOP + 专用工具 + 历史案例库)

**⑦ Multi-Agent(质量层)**
- 驱动力:单 Agent 三大困境——上下文膨胀(越聊越忘)、注意力分散(易跑偏)、串行瓶颈(任务排队慢)
- 设计:七种模式——主从委托/接力/对等讨论/层级/竞争投票/评估反馈/动态路由,生产中常组合使用
- Why:收益 = 上下文隔离 + 并行 + 专家化 + 交叉验证;但**成本翻 N 倍** → 先单 Agent + 多 Skill,扛不住再拆
- PM 类比:把全能员工拆成项目组;"能一个人干完的别开会"。映射:手册 Skill Router(克制版)
- PM 拍板:忍住「不到必要不拆」,默认单 Agent + 多 Skill
- 案例⑦:若升级为"排查 + 近 7 天失败趋势分析 + 写对外回复"才拆排查/分析/写作三专家 + 协调者

**⑧ Harness(质量层,★ 枢纽)**
- 驱动力:没保护跑不过 24 小时;**LLM 工具格式错误率 5-10%,幻觉更高**;Demo 能跑 ≠ 生产能用
- 设计:运行时护甲,六大子系统——错误分类与恢复 / 上下文工程 / 迭代控制 / 工具治理 / 安全防护 / 可观测性
- Why(★):**从"能跑"到"能用"的分水岭**;不改变 Agent 决策,只让它活得久跑得稳
- PM 类比:Agent Loop 是大脑,Harness 是免疫系统 + 骨骼 + 皮肤;手册的 Audit、工具治理、人工审核、容错重试合起来 = 企业版 Harness。映射:手册 Audit #11 / Tool Governance / 人工审核 #12
- PM 拍板:上线红线三件套——**可观测 + 高危人工审核 + 工具权限,缺一不可发布**;详见配套《Harness 安全底座》专题图
- 案例⑧:日志接口超时→自动重试;反复调挂掉接口→迭代上限掐断;想调"重置接入方密钥"→挡下转人工;全程留痕且 appsecret 不落日志

**⑨ Claw(形态演进)**
- 驱动力:服务端 Agent 只能"告诉你怎么做",用户还得手动执行
- 设计:Agent 以本地客户端运行,直接读文件、跑命令、改代码、提 PR
- Why:闭合「决策-执行-验证」,消除"说"和"做"的断层;本质同服务端,核心是跑在本地
- 定位:不是新原理,是**部署形态**演进——"远程顾问"变"贴身秘书";权限、安全要求更高
- PM 拍板:评估要不要放到用户本地、权限怎么收;能力越靠近真实系统,出事代价越大
- 案例⑨(收束):Agent 在对接同学电脑上直接定位签名拼接代码、改好、验证、提 PR——这一单才算真正办结

### 贯穿案例一页回看(02·)
1 LLM 通用套话 → 2 记忆记住 V2 → 3 RAG 查内部文档没依据明说 → 4 Function Call 调日志发现超时 8 分钟 → 5 Loop 串多步定位双重病因 → 6 Skill 固化 SOP → 7 Multi-Agent 按需拆专家(克制)→ 8 Harness 容错/掐循环/高危转人工/留痕 → 9 Claw 本地改代码提 PR 办结。

### 暗线:控制权的钟摆(03)
早期·全用规则(死板)→ 自主 Agent·全交 LLM(灵活但失控、会崩)→ 工程实践·收回代码/流程(可控、可复盘、稳定)。最终收敛于**「LLM 负责判断、代码负责决策与兜底」= deterministic-picker**,即"概率系统嫁接在确定性工程上"的具体落地。

### 带走的两句话 + 落地忠告(04)
- 总答案:九阶段归三层(知识/能力/质量),每个模块都是被"上一代的痛"逼出来的补丁,没有一个是炫技
- 手册的历史定位:用户的手册不是"一种选择",而是演进史验证过的**收敛结论**的企业落地版
- 忠告 1:**观测比开发更重要**——Agent 是概率系统,模型升级一版链路可能悄悄退化;先建可观测体系(轨迹录制、成功率、badcase 归因),持续打磨是常态
- 忠告 2:对前沿架构保持敏感——领域半年一代际;好架构思路(渐进式加载、结构化压缩)比堆人力调参有效 10 倍

## 关键细节(量化数据)

- 50 个工具说明书 ≈ 5 万 token;Skill 摘要 ~100 token/个;渐进式加载 = 10 倍 token 效率
- 生产 90% 场景用 ReAct 主循环 + 按需 Plan
- 案例量化:timestamp 偏差 8 分钟 vs 5 分钟有效期;V2 签名 = 字典序拼接 + HMAC-SHA256
- 枢纽阶段:④ 决策执行分离、⑤ Loop、⑧ Harness(三个 ★)
