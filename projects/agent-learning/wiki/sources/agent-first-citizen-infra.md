# 重新思考研发基础设施：当 Agent 成为第一公民 — 精读笔记

> TL;DR：软件研发从来都是"意图(不确定性)驱动 + 代码(确定性)沉淀"的进化体，Agent 没有改变这个模式，只是把"意图→代码"循环从周/月级压到分钟级；这导致为人设计的 developer infra(Git/CI/CD/CR 的全部慢循环假设)系统性失配，必须按"可理解、可操作、可感知、可追溯"四层重建为 Agent-Oriented infra——Agent 的自主程度不取决于它多聪明，而取决于 infra 提供了多强的安全护栏。

## 核心论点

### 一、为什么要重新思考(诊断)

**1. 统一框架："意图驱动 + 代码沉淀"(p2-3)**

无论传统研发还是 agent 研发，软件系统一直是"意图(不确定性)驱动 + 代码(确定性)沉淀"的进化体。模式从未改变，改变的是驱动和沉淀的速度与机制。

- 传统循环：用户意图 → 产品经理压缩意图 → 研发翻译成代码 → 上线 → 新意图。人是意图到代码的桥梁，桥梁带宽决定循环速度(周/月级)，受人的认知带宽物理约束。
- Agent 循环：意图(SKILL.md / 自然语言) → agent 生成代码 → 执行 → 反馈 → 修正意图。桥梁从人变成 agent，循环压到分钟级。

**三个推论(p3-4)：**
- 推论一：Agent 不是革命，是加速。没有改变软件进化的基本结构，只把循环速度提了几个数量级。因此不需要推翻已有认知框架，只需追问：速度变化后，哪些慢速假设不再成立？
- 推论二：静态沉淀不会消失。不确定性被消除后，确定的逻辑应该用确定的方式表达——"用一个巨大的概率模型去逼近一个已经确定的函数，在信息论意义上是荒谬的浪费"。agent 系统成熟后会把验证过的逻辑固化为代码，agent 占比下降；新意图涌入时再上升。
- 推论三(本文主旨)：模式没变，但模式对基础设施的要求彻底变了。循环从月级到分钟级后：Git 假设每次变更都值得 commit；CI/CD 假设构建部署是离散事件；测试环境假设代码稳定可提前准备好环境；发布系统假设发布低频、需人工审批;CR 假设有人看每一行代码——这些假设在慢循环下全部成立，在快循环下全部失效(p4)。

**2. Agent 占比的"锯齿形"曲线(p3-4)**

把系统中 agent 占比画成时间曲线应呈锯齿形：意图涌入时陡升(不确定性高，agent 探索)，逻辑固化时缓降(代码沉淀)。两条典型路径：
- agent-native 系统(周报系统)从接近 100% 高位出发，connectors.yaml、CI 配置、check_connectors.sh 逐步沉淀，占比下降；新意图涌入又上升，但每次上升幅度递减(骨架越来越稳)。
- 存量系统(镜像服务)从低位出发，agent 逐步接管，每次接管是一个小锯齿(生成配置上升、审批入库固化下降)，整体趋势爬升。
- 诊断工具：如果曲线是平线，说明有问题——要么需求没有真的持续变化(健康存疑)，要么该做的代码固化没做(不健康)。锯齿形本身是系统健康度诊断工具。

**3. 三个变量的变化(p4-5)**

Agent 改变了统一框架里三个关键变量，三者同时变化的效果是**乘法而非加法**：
- **桥梁带宽**：从人到 agent，提升几个数量级。人的角色从"桥梁"后退为"意图的表达者 + 结果的验收者"。Agent 优势在认知带宽(不丢上下文、不漏步骤、不混配置)，与"聪明"无关，Standard 级别模型就够。
- **沉淀粒度**：代码分化为两种形态。**瞬态代码**——on the fly 生成的一次性中间产物，分钟级生命周期，用完即弃，不提交 Git(类比 JIT 编译器生成的机器码)。**沉淀代码**——经过验证、反复出现、固化下来的逻辑，仍需版本控制与测试。反直觉规律：越是动态的系统，越需要坚固的静态约束——类型系统、沙箱、权限边界成为 agent 系统可靠运行的护栏(p5)。
- **循环频率**：从发布周期到运行时反馈。迭代的性质变了——从离线的、批量的、有审批流的变更，变成在线的、增量的、实时反馈的演化。系统从离散事件变成连续过程，"发布"概念本身变得模糊(p5)。

**4. 推到极致：意图可枚举假设崩塌(p6)**

传统软件的根本假设是"意图可枚举"(PRD 把无限意图空间压缩成有限 feature 集合)。当意图多样性和变化速度超过发版频率，管线在架构上就不支持了。agent 还会承担产品经理的部分职能——运行时直接理解用户意图并生成执行路径；产品经理从"意图的唯一入口"变成"意图的策展人"。面对开放意图空间，agent 占比有一个不可压缩的下界(因为不知道明天用户会提出什么)。此话题属 Product Infra 范畴，本文不深入。

### 二、具体怎么改(处方)

**根本判断(p8-9)：Agent 的自主程度是 infra 安全能力的函数。**
- 传统 infra 是 People-Oriented：安全靠人的自我约束 + 事后审计。设计假设：操作者有常识、有责任心、操作频率低、能从口口相传的隐性知识中填补概念裂缝。Agent 不符合任何一条：会 hallucinate、操作频率极高、出错需自动恢复、面对概念裂缝会卡住或猜错(p9)。
- Agent-Oriented：安全靠机制化保证——资源归属、权限管控、dry-run、分级策略、自动回滚。本质是把安全模型从"依赖人的自我约束"重建为"依赖 infra 的机制保证"。**给 infra 补能力，而非给 agent 加限制。Infra 的安全能力越强，agent 的自主空间越大。**
- 核心抽象：面向 agent 的 infra 设计，就是把原来隐式依赖人的东西——认知、常识、责任心——显式化为系统机制(p9)。

**四层设计原则(p10)，递进关系：先理解 → 再操作 → 感知反馈 → 全程可追溯，每层覆盖 API 接口设计和运行时环境两个维度：**

1. **可理解(Comprehensible)**——agent 能建立正确的心智模型。概念体系必须自洽、完整、不依赖口口相传的隐性知识。"复杂度不是问题，不一致才是"：一个概念自包含的 CLI，即使 200 个子命令、每个 30 个参数，agent 也能多轮试探快速学会。运行环境也要自描述(manifest / devcontainer 声明依赖、输入输出、允许工具)，这是 harness 初始化的前提。
2. **可操作(Operable)**——agent 能安全可靠地行动。关键能力：可试探(dry-run / preview / 幂等重试)、操作原子可组合(输入输出类型化)、隔离执行(每任务一个 sandbox，试错代价可控)、凭证不进 sandbox(通过 vault / egress proxy 注入能力，agent 不持有明文凭证)、渐进信任(低风险 agent 自主执行，高风险准备好变更计划 + dry-run 结果、人来点按钮)。回滚能力把不可逆操作变成可逆操作，等于降低风险等级。
3. **可感知(Observable)**——infra 把状态和结果清晰地交回 agent。"沉默和含糊是 agent 的敌人——Unix 的 'Silence is golden' 对 agent 有害"。状态必须 API 可查、结构化、实时;每次响应都要提供足够语义信息让 agent 决定下一步;结果要可判定(产出转成 CI 能识别的 pass/fail，必要时保留人工 review gate)。
4. **可追溯(Traceable)**——过程不丢失、可恢复、可回放。状态可恢复(snapshot / checkpoint，避免长任务因容器故障从零开始)、过程可回放(至少保存 artifact，更进一步保存 execution trace，可以 fork/replay)。附加维度：**通过 agent 的行为观测 infra 本身的设计质量**——agent 反复重试某 API → 反馈语义不足；调用顺序混乱 → 前置条件没有显式化；用错身份 → 身份体系碎片化。人遇到这些问题会默默绕过去(问同事、手动修、记住坑)，不产生可观测信号;agent 每次都忠实撞上去且频率极高——agent 的失败模式是 infra 设计缺陷的放大器，把 infra 设计质量从主观判断变成了可度量的东西。

**行动方向(p12-13)：**
- **Harness 平台化**：把 agent 运行环境从"每个团队自己在云服务器上搭"变成平台内建能力。Harness spec 声明式定义(角色、工具、凭证、workspace、skill)，handoff 时自动初始化、完成后自动回收。核心约束是即时供给——并行 agent 数量对环境供给是乘法压力，初始化时间直接决定并行效率。本质是云原生/serverless 思路，组装对象从"应用"变成"agent 工作环境"。
- **统一身份体系**：agent-native 身份——不是借用人的 token，而是 agent 拥有自己的机器人身份，一个身份在所有 infra 上通用，自动轮转、最小权限、操作可追溯到具体 agent 实例和意图。身份与 harness 联动：初始化时注入角色对应凭证，不同角色天然隔离。远期方向是 credential brokering——agent 能使用凭证但不持有凭证。
- **Dry-run 基础设施**：在基础设施层提供统一的"变更预览"能力，而非每个工具各自实现。资源归属治理是 dry-run 的前提(没有归属关系没法计算影响范围)。Dry-run 输出应标准化：diff + 影响范围 + 影响规模 + 风险等级 + 是否可回滚。这是渐进信任机制的基础——根据 dry-run 输出的风险等级决定 agent 自主执行还是交给人审批。"有 dry-run 就放心"应被泛化为一种 infra 能力，覆盖所有生产环境资源变更。
- **验证体系的演进**：传统软件测试和 agent 评测从两端走向融合——传统测试验证"代码是否正确实现了意图"，agent 评测验证"模型是否正确理解了意图"，当系统是 agent + code 混合体，两者融合成一件事：从意图到结果的端到端验证。融合形态特征(p13)：断言方式从精确匹配(`output == expected`)演进到约束满足(输出满足一组约束——类型正确、不违反安全策略、业务逻辑自洽，具体实现路径可以不同)；确定性验证与概率性验证分层共存(同一测试中确定性部分用传统断言，概率性部分用 LLM-as-judge 或统计方法)；测试从独立阶段变成持续验证——CI 管固化代码(低频、确定性)，嵌入式验证运行时管 on the fly 代码(高频、概率性、约束验证)。
- **判断：验证基础设施的投资优先级应该高于生成能力**(p13)。生成能力的提升是模型厂商在推的事，验证能力的提升是 infra 团队该做的事——验证的可靠性直接决定了 agent 的自主空间。
- **Agent 行为驱动的 Infra 质量度量**：通过 agent 的行为模式(重试频率、错误类型、绕路路径)来度量 infra 的 agent 友好程度。agent 成为 infra 设计的持续压测者，用得越多暴露的设计缺陷越多。

**结论(p13)**：今天的 Git、CI/CD、Code Review 不会消失，但管辖范围会收缩到"持久化代码"的领域——而那个领域的面积在缩小。真正需要被建设的，是支撑 agent + code 动态混合体的新一代基础设施。方向：给 infra 补能力，Infra 的能力边界就是 agent 的自主边界。

## 关键细节与数据(全部量化数据)

### 引子案例：周报系统(p1-3)
- 架构：connector 打通异构系统(代码托管、项目管理、数据平台、IM) → 规范化成统一活动流 → 用一个意图(skill, `SKILL.md`)驱动聚合与排版 → 生成单文件 HTML → 推送到内部 Pages。connector 是作者自写的一组 CLI 小工具。
- 三类组成：agent(核心驱动，读 SKILL.md)、代码(大部分 agent 每次运行时按需生成、生命周期极短)、infra(Pages 托管、CI 部署、Git 版本控制——但对作者"不可见")。
- Agent 占比约 90%(p3)，意图→执行循环周期为分钟级。代码生命周期从"月/年"缩短到"分钟"(类比 JIT 编译)。
- 一轮迭代中意图改了多次(加会议决策模块、加交叉洞察、调整模块顺序、改标题、overview 加关键贡献者)，每次改的是 skill 而非僵死的程序，改完 agent 立即重新生成代码并部署，整循环分钟级。若走传统产品经理→研发→CR→CI 管线，每次可能是半天迭代周期，全部做完可能要两周(p2)。
- 有趣副产品：每加一个 connector 等于给团队定义了一种工作模式(如 `cli-meeting-minutes` 指向内部 IM 某文件夹，人把会议纪要放进去 connector 才有数据)——系统对人的要求从"按流程执行"变成"按规范沉淀"(p3)。
- 沉淀代码实例：`connectors.yaml`(数据源定义)、CI 部署配置、`check_connectors.sh`(覆盖体检脚本)(p5)。

### 引子案例：镜像仓库建站运维(p2-3, 4)
- 跨 6+ 个异构平台、涉及 12+ 个配置文件的结构化 SOP，为新地域搭建镜像仓库托管链路。传统排期 2 人日，AI 辅助后 1–1.5 小时完成(p2)。
- Agent 占比约 50%(p3)：agent 执行全部技术操作(API 调用建仓库、配置同步策略、生成十多个配置文件、提交 CR 并推送)；人处理风控环节(生产环境配置推送和白名单变更)——技术上完全可自动化，但错推可能导致线上故障且缺乏灰度回滚能力，所以留给人。**人机边界由出错代价决定，由风控机制决定**(p3, p5)。
- 人易出错细节：不同环境密钥加密实例不同、密文不可互换;不同 region 镜像同步走不同流水线;缓存要刷新才生效——细节散落 SOP 各角落，人需全程高度注意力。结果是"新手 + AI = 老手的产出"，SOP 从"面向人的菜谱"变成"面向 agent 的可执行意图"(p4)。
- 中间形态：agent 每次建站生成的配置代码不是用完即弃——提交 Git、走 CR、发版上线，但生成过程是 on the fly 的，人审批后固化(p5)。

### 案例一：多角色 agent 研发系统的痛点(p6-7)
基于内部研发 CLI 搭建多角色 agent 系统，云服务器上运行 discovery(需求理解)、delivery(代码开发)、examiner(Code Review) 等角色，覆盖需求理解到发布上线全链路。做得不错的：研发 CLI 结构化 JSON 输出(agent 特别喜欢用 `--format json`，哪怕没要求也主动加)、统一命令体系覆盖研发全流程、skill 文档 + `cli -h` 基本能让 agent 自适应、统一鉴权框架统一多 CLI 身份。痛点六条：
1. **运行环境全靠手工搭建**：一台云服务器同时维护多模型入口、多套 CLI、多个 agent profile、多套凭证和代理配置;多 agent 并行时权限容易串、环境容易乱。delivery 角色需要的"完整工作环境"包括：任务 workspace 和目标 repo、前序环节(discovery/standards)产出的 artifact 和上下文、本任务允许的工具和外部副作用范围、隔离的凭证和身份配置、对应 skill 和 prompt、沟通和 handoff 通道——远超"沙箱"概念，作者称之为 **harness**。
2. **研发规范没有机器可读的稳定承载**："这个应用到底该怎么研发、测试、发布"(Dockerfile 怎么写、测试命令、发布路径走日常→预发→正式还是项目环境→预发→正式、CR 和 pipeline 怎么创建)散落在人的经验、历史任务、平台页面、仓库文档里;agent 只能临场猜或靠 prompt 硬编码。
3. **接口带有偏"人用"的痕迹**：MR 的全局 id 和项目 iid 缺少强类型化区分，agent 经常传错;隐性规则散落口头知识("回复 MR 评论必须回复 root 评论，回复在子评论下面是看不到的");`--dry-run` 只停留在研发 CLI 层，无法穿透到底层平台;pipeline 报错或超时后 agent 不知道下一步该干什么——状态机的边界对 agent 不显式。
4. **验证的难度梯度极大**：CLI 验证简单(输入输出确定性高)，app 验证困难(UI 交互和状态)，web 验证超级困难(浏览器环境、异步渲染、用户交互路径爆炸)。目前只能靠人打 skill 补丁勉强解决 app 验证。
5. **Code Review 缺上下文**：agent 产出 MR 数量爆炸式增长后，reviewer 不知道需求是什么、为什么这么改、哪些能力必须保留、MR 在系统设计里承担什么角色，就只能看表面问题(代码风格、命名、是否有测试)；真正危险的缺陷在更上层：需求理解错、scope 漏、方案方向错、破坏系统整体设计。Review 同样需要 harness——自动准备原始需求、discovery 产出的上下文和验收标准、研发规范、MR diff 加 CI 状态、本次变更影响的上下游。
6. **没有可重复的实验场**：很多验证只能拿真实仓库、真实 MR、真实 CI 来跑——生产任务变成了集成测试。需要可重复创建和关闭 MR、可控 CI pass/fail、可控 reviewer approve/reject 的实验场，或一套贯通所有环节的 dry-run。

### 案例二：配置推送(p7-8)
- 同事不敢让 agent 推送配置，顾虑：配置推送的权限管控几乎没有——agent 逻辑上可以误推任何配置，包括其他高风险业务的配置。这是对 infra 安全能力缺失的精确识别，跟"不信任 AI"无关。
- 假设给出强 dry-run：`config-tool publish data=xxx.json --dry-run` 输出 `diff / affected apps / affected pods: about 1,024`,同事回答"如果是这样，就比较放心让 agent 自己推送配置了"。
- 瓶颈不在 agent 侧(能力足够)，在 infra 侧：没有资源归属治理(配置属于哪个应用/团队/环境，无机器可读的归属关系)、没有 dry-run(推之前不知道会影响什么、影响多大)、没有分级策略(影响 3 个 pod 和 1,024 个 pod 走同一条路径)、回滚能力不足(推错了没有秒级回滚)。
- 可推广判断：所有生产环境资源变更(应用发布、数据库 migration、网络策略、DNS 修改)在"资源归属 × 影响可预知 × 渐进信任 × 可回滚"四个维度上安全保证强度不同，agent 的自主空间也就不同。回滚能力把不可逆操作变成可逆操作，等于降低风险等级——这是 infra 能为 agent 做的最有杠杆的事之一。

### 案例三：身份与鉴权碎片化(p8-9)
- Agent 使用基础设施时最常遇到的阻塞是"身份不对"——比"不会用"更频繁。
- 公司统一鉴权框架统一了多个 CLI 的身份，但 git 仍是独立 SSH 体系。对人是一次配好就行;对 agent 是两套凭证管理逻辑、两套轮转机制、两套报错模式——harness 初始化、错误排查、多角色配置的复杂度都翻倍。
- 沙盒环境更尖锐：agent 7×24 无人值守运行时传统鉴权全部失效——Web SSO 需要浏览器(沙盒没有)、短期 Token 2-8 小时过期(凌晨 3 点 token 过期 agent 直接卡死、需人工介入刷新)。解法是长效 Private Token——一次生成、长期有效、完全 headless、后端零侵入。方向："从以人为中心的短会话，向以 token 为中心的长效凭据演进"(引自内部实践,飞桨《沙盒环境 Agent 7×24 小时工作，免登鉴权怎么破?》, 2026-05-26)。
- 即便如此仍是多套身份体系并存：统一鉴权 for CLI tools、SSH for git、Private Token for 代码平台 API,每套单独配置、单独排错。
- 结构性解释：**角色分工曾经屏蔽了身份碎片化的复杂度**。人分前端/后端/运维/设计/数据等角色，每人实际只接触自己角色对应的 2-3 套身份系统;agent 打破角色边界(一个 delivery agent 要 git clone(SSH)、研发 CLI 提 MR(统一鉴权)、操作中间件配置(中间件身份)、查监控(监控平台身份))，身份体系数量从"每人 2-3 套"膨胀到"每 agent 5-8 套"(p8)。
- 第二层放大效应是频率：人一天切换身份体系几次、摩擦被工作间隙吸收;agent 一分钟内可能连续调 git → 研发 CLI → 中间件 → 监控。**两个乘法因子：角色融合导致身份体系数量膨胀 × 工作频率导致切换次数膨胀**(p9)。理想状态：agent 拿到一个身份，在所有 infra 上都能用。
- 三个案例共性(p9)：具体问题各不相同(运行环境、接口设计、资源变更安全、身份鉴权)，根源同一个——**这些 infra 都是为人设计的**。

### 行业趋势(p11-12)
- **并行 agent 需要并行 infra**：Superset(面向 multi-agent 开发的 IDE)每个开发者同时跑最多 12 个 coding agent，三人团队每天产生约 600 个 preview deployment，平均构建时间约 30 秒(ref: Vercel Blog, 2026-05-10)。关键洞察：底层 infra 任何一层慢，上层并行性崩塌——12 个工作流压缩成一个队列，几分钟的任务变几小时。验证两件事：(1) harness 供给必须即时——"环境初始化需要 5 分钟"在 12 个并行 agent 面前就是 60 分钟等待;(2) agent 数量对 infra 压力是乘法关系：1 开发者 → 12 agent → 12 套隔离环境 → 12 条并行 pipeline → 12 个即时部署，infra 吞吐量必须线性 scale with agent 数量。三人团队没有 platform engineering，整个 infra 靠平台即时供给能力支撑——暗示面向 agent 的 infra 不应该需要专门团队维护环境管理。
- **Human DX 和 Agent DX 是正交的**：Google 的 Workspace CLI(gws)从第一天就按 agent-first 设计(ref: Justin Poehnelt, "You Need to Rewrite Your CLI for AI Agents", 2026-03-04)。精确区分：Human DX 优化的是可发现性(discoverability)，Agent DX 优化的是可预测性(predictability)。关键实践：schema 自省替代静态文档(`gws schema drive.files.list` 输出完整方法签名，CLI 成为 API 运行状态的权威来源)、输入加固防 hallucination(路径沙箱化、拒绝控制字符、资源 ID 校验——因为 agent 会 hallucinate 出 `../../.ssh` 这样的路径穿越)、发布 SKILL.md 文件编码 agent 无法从 `--help` 推断的不变量("写操作必须先 dry-run"、"list 调用必须加 --fields")、响应消毒防 prompt injection。关键观点：**agent 不是可信的操作者(The agent is not a trusted operator)**——会 hallucinate、生成路径穿越、在资源 ID 里嵌入查询参数;面向 agent 的设计需要把 Unix 哲学里"Trust the user"的假设翻转过来(p12)。
- **Credential Brokering 行业趋同**(ref: Tony Dang, Infisical Blog, 2026-05-23)：agent 需要凭证访问外部服务，但 agent 本身不能被信任持有凭证(prompt injection 可能导致凭证泄露)。解法是在 agent 和凭证之间加一条信任边界——credential broker(代理层)：agent 发请求时携带占位符(如 `__github_token__`)而非真实凭证，broker 认证 agent 身份后将占位符替换为真实凭证转发请求，agent 全程见不到真实凭证。Anthropic(Managed Agent Infrastructure)、Vercel(Sandbox 凭证注入)、Cloudflare(Outbound Workers)、LangChain(Sandbox Auth Proxy)各自在不同层实现了同一范式。案例三的身份演进路径——短会话 token → 长效 Private Token → credential brokering——是一条从"agent 持有凭证"到"agent 完全不接触凭证"的渐进线;多家公司独立趋同同一范式，说明这是 agent 安全模型的必然走向。

- "复杂度不是问题，不一致才是。"(p10)
