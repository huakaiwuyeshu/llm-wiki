## 核心内容

### 案情
接入方技术对接群发来:「你们的**回调签名验证一直失败**,接口对接卡住了,帮忙看下?」——手册「联调排查」链路最典型的一单。看从只会聊天的 LLM 到能把这单办结的 Agent,每阶段补了什么。

### 九步成长(每阶段:发生了什么 / +能力 / −仍卡在哪)

**1 LLM(只会聊天)**
- 发生:裸 LLM 泛泛而谈「签名失败通常是密钥不对、参数排序错误、时间戳过期……建议逐一排查」——话没错,但不知道我们的签名算法、接入方是谁、日志在哪,等于网上随处可见的通用答案
- +:理解人话、生成通顺回答 | −:不懂业务、不能查、没记性,只能给套话

**2 记忆(记得住对话)**
- 发生:技术支持追问「他们用的是 V2 签名,按这个继续」;Agent 记住"这一单是 V2",后续不必反复重申;也记得该接入方上周问过回调地址
- +:跨轮次、跨会话记住上下文 | −:记住"是 V2",但 V2 算法细节仍不知道

**3 RAG(查得到内部文档)**
- 发生:检索内部《签名规则文档》,捞到 V2 具体步骤(**参数字典序拼接、HMAC-SHA256、时间戳 5 分钟有效**),答「请先确认拼接顺序;时间戳超 5 分钟会失效」;文档没有就明说"未找到依据"而不编
- +:回答基于真实文档,有据可查 | −:能讲规则,没法真去看这一单的失败日志

**4 Function Call & MCP(能动手调接口,★ 枢纽)**
- 发生:输出调用指令(日志查询接口 + 该接入方 appKey + 时间段),代码侧执行捞回日志,发现「**提交的 timestamp 比服务器晚 8 分钟,超过 5 分钟有效期**」;决定"查日志"的是 LLM,真正执行的是代码——这道分离是后面所有安全的根
- +:从"只会说"到"能调真实接口拿真实数据" | −:单次调用够了,但这单要查日志→对参数→比时间好几步

**5 Agent Loop(自己串多步,★ 枢纽)**
- 发生:自跑排查循环:查日志→见时间戳异常→决定再核参数拼接→调接口比对→确认拼接也有问题→汇总;"查了第一步才知道第二步查什么" = ReAct「想→做→看→再想」。结论:「**两个问题:① 时间戳超时(主因);② 参数 `amount` 没参与签名。修复:校准服务器时间 + 把 amount 纳入拼接**」
- +:多步排查自动串成链,边查边调方向 | −:这单会查了,但"续租失败""号池耗尽"等几十种问题每次都从零摸索

**6 Skill(按 SOP 精准排查)**
- 发生:把"签名失败排查"做成 Skill(SOP + 专用工具 + 历史案例库);识别出签名问题就加载能力包,按固定步骤排查而非临场发挥;平时只看一句"我会排查签名问题"的摘要,命中才展开完整手册——省大量上下文
- +:老师傅经验固化成可复用 SOP,又快又稳又省 token | −:单 Agent 同时背排查、分析、写报告多角色会"精神分裂"

**7 Multi-Agent(专家分工,按需)**
- 发生:假设升级——还要同时分析该接入方近 7 天失败趋势 + 写对外回复;拆成排查专家 + 数据分析专家 + 写作专家 + 一个协调者;但只有真扛不住才拆,单这一单其实一个 Agent + 多 Skill 就够
- +:复杂任务拆给多专家并行,各管一摊 | 警惕:拆一次成本和协调复杂度翻倍,不到必要别拆

**8 Harness(上线护甲,★ 枢纽)**
- 发生:真实流量的意外——日志接口偶发超时(自动重试)、模型反复调挂掉的接口(迭代上限掐断)、想调"重置接入方密钥"高危操作(挡下转人工审核)、排查轨迹全程留痕可复盘(且 appsecret 不落日志);护甲不改变怎么排查,只保证活得久、不闯祸、出事查得到
- +:从"demo 能跑"到"敢上生产"——容错、限流、审核、可观测 | −:仍在服务端,给出修复建议后还得人去操作

**9 Claw(本地端,直接动手)**
- 发生:Agent 跑在对接同学电脑上,直接打开代码、定位拼接签名那段、改好、跑验证、提 PR——而不是只甩一句"请把 amount 纳入签名";"说"和"做"的断层被消除
- +:闭合决策→执行→验证,"远程顾问"变"贴身秘书" | 代价:能力越贴近真实系统,权限边界和安全要求越重

### 一页回看
1 通用套话 → 2 记住 V2 → 3 查文档讲真实规则、没依据明说 → 4 真调日志接口发现时间戳超时 → 5 自串多步定位双重病因 → 6 排查打法固化 SOP → 7 复杂时按需拆专家(克制)→ 8 容错重试/掐死循环/高危转人工/全程留痕 → 9 本地改代码、验证、提 PR,闭合执行。

## 关键细节
- 案例量化:V2 签名 = 参数字典序拼接 + HMAC-SHA256 + 时间戳 5 分钟有效;实测 timestamp 偏差 8 分钟;双重病因 = 时间戳超时(主因)+ `amount` 未参与签名
- 业务问题域举例:签名失败、续租失败、号池耗尽、回调地址(印证 API 接入/号池类业务)
- 高危操作示例:重置接入方密钥;敏感数据示例:appsecret 不落日志
- 阶段间连接词链:想让它懂签名规则→规则在内部文档→得真去查日志→多步谁来串→沉淀成固定打法→复杂工单要协作→上线扛得住意外吗→直接动手而非只给建议

## 洞见
