# Agent 核心技术概念与范式演变及背后的思考(精读笔记)

> TL;DR:阿里技术工程师"飞樯"复盘 2023–2026 年 Agent 范式的四阶段演进(被动式 ReAct → Workflow Agent → 自主 Agent → 自进化 Agent),并沿 Prompt / Planning / Memory / Tools / Workflow / Environment 六大维度逐一对比前后变化,核心结论是"形未变、神已变"——经典模块框架依旧,但每个模块的实现范式都被重构,底层思想是"通过工程化手段构建确定性,以承载模型不确定性"。

## 写作动机与核心论点

- 动机:Claude Code、Codex、OpenClaw、Hermes 等新一代 Agent 产品/框架涌现,Agent 能力出现跃升;但社区学习材料新旧概念混杂,很多人读到的是早期文章,产生"穿越感"(p1)。
- 核心论点一:Agent 技术演变前后并非简单替代关系,而是相关、继承、甚至可相互结合的关系;新旧范式**并存且互补**(p1, p4)。
- 核心论点二:不理解范式背后的演化逻辑,容易陷入"为了升级新架构而升级架构"或"盲目追求最新技术概念"的误区;本文目标是回答"Agent 技术是怎样变化的?为什么会这样变化?"(p1-2)。
- 核心论点三(总结处):宏观架构上今天的 Agent 仍由 Prompt、Planning、Memory、Tools 等经典模块组成,与 Lilian Weng 早期理论框架并无二致——**"形"未变,但"神"已大不同**,是一场"内核重构"(p12)。
- 方法论主线:Agent 正在从"魔法调优"走向"系统工程";不再依赖模型"智商"硬扛所有问题,而是用更精细的工程化手段(文件化解耦、CLI 原生利用、沙箱隔离等)弥补模型不足、放大模型优势(p13)。

## 演变脉络一:Agent 发展的四个阶段(2023–2026)(p2-4)

作者强调:四阶段并非线性平滑过渡,也非完全替代,而是并存互补;落地时应按业务复杂度、对稳定性的要求、成本预算选择范式或组合使用(p2, p4)。

### 阶段一:早期 Agent(被动式 ReAct)— 2023(p2-3)

- 2023 是 LLM 爆发元年、Agent 概念的"启蒙期"。代表理论:Lilian Weng 博客《LLM Powered Autonomous Agents》,定义基本架构 **LLM + Planning + Tools + Memory**;同期开源项目:AgentGPT、AutoGen、MetaGPT(p2)。
- 本质是"被动式响应":核心架构是初步的 **ReAct(Reasoning + Acting)**,单步"Reasoning → Observe → Response"链路;受限于基础模型效果,能做好 3 轮以上 Reasoning 的模型都不多(p3)。
- 特点:交互形态类似增强版 Chatbot("一问一答"/"指令-执行");能力边界严重依赖明确指令,虽有 CoT(Chain of Thought)与简单 Function Call 链条,但基本只能完成单点、短链路小任务;局限性是缺乏长期规划能力,任务复杂度超出上下文窗口或逻辑链条过长就偏离/中断(p3)。

### 阶段二:工作流 Agent(结构化与可控性)— 2024(p3)

- 背景:to B 业务对稳定性要求提升,纯 ReAct "理想方式"解决不了复杂问题,**Agentic Workflow** 成为主流。核心理念:**用工程化的约束来弥补模型的不确定性**。代表:LangGraph、Dify 等 Workflow 编排(p3)。
- 架构特征:要么整个框架是固定 Workflow、**关键节点嵌入 LLM**;要么 LLM 作为中枢调用预定义子 Workflow。作者将其类比为早期 Harness("驾驭工程里的约束")——当时虽无此概念,目标本质相同(p3)。
- 应用场景:to B 领域极受欢迎,企业服务/日常重复性工作不需要真"智能决策",只需按步骤 1、2、3 按时、按量、保质完成(p3)。
- 价值:对非长尾、非极度复杂场景,Workflow Agent 至今仍是性价比最高、落地最稳定的方案,能确保效果下限(p3)。

### 阶段三:自主 Agent(复杂规划与长程任务)— 2025(p3-4)

- 作者认为 2025 是 Agent 迈向"自主性"的关键转折点:Manus 为代表的通用 Agent 火爆,Claude Code、Codex 等 AI Coding Agent 出现;2026 年初火爆的 OpenClaw 等框架继续扩大受众,巩固趋势(p3)。
- 特征(Autonomous Agent):
  - 核心变化:不再满足于"调用几个工具后给结论",具备复杂 **Planning** 能力,面对模糊/宏大需求可自行拆解任务、规划路径、调用工具、多轮迭代(p4)。
  - 长程任务能力:用户清晰描述需求 + 设定好开发规范(Specs),Agent 可连续运行很长时间,自主处理企业级项目代码或复杂业务流程(p4)。
  - 自我校验:配合**轻量级 Harness 或自我校验机制**,模型能在长程运行中不断修正错误,交付高质量结果——从"辅助者"向"执行者"的根本角色转变(p4)。

### 阶段四:自进化 Agent(持续学习与自我升级)— 2026(p4)

- 背景:2026 年 Hermes Agent 等新一代框架兴起,配合 LLM-Wiki 等开源项目,Agent 可以自我沉淀 Skill、自我沉淀知识库,甚至通过 RL 训练提升模型能力,进入 **Self-Evolving(自进化)** 阶段(p4)。
- 核心本质:开始解决"静态模型"与"动态世界"之间的矛盾(p4)。
- 机制原理:Agent 不仅完成任务,更在过程中**沉淀经验**;通过记忆模块、反馈循环和自我反思机制,将从前一次任务获得的教训转化为新知识或策略(p4)。
- 最终目标:实现"**越用越好用**"——根据历史交互数据自动优化自身提示词、工具选择策略,甚至微调局部模型参数,实现自我升级和进化(p4)。
- 意义:Agent 从"一次性消耗品"变成"**可积累资产**",为构建真正具备长期生命力的数字员工奠定基础(p4)。

整体演化主线:从"简单交互"到"复杂执行",再到"智能成长"(p4)。

## 演变脉络二:六个核心技术概念的前后变化(p4-12)

引子:今天创建哪怕最轻量级的 Agent,除了最关键的 Agent Loop,还涉及 Prompt、Planning、Memory、Tools、Workflow、Environment 六个维度(p4)。

### 1. Prompt:深耦合 → 渐进式加载(p4-6)

- 早期:"一个任务创建一个 Agent"(如写作 Agent + 编辑 Agent + 绘图 Agent 协作产文),每个 Agent 背后是一段精心调试、独立存在的 System Prompt(含人设、任务目标、要求、约束、注意事项、示例)。Prompt Engineering 几乎等同于对每个任务单写一段"小作文",维护成本极高,场景增多后 Prompt 管理极混乱(p5)。
- 瓶颈:"系统级指令"与"任务要求&细节"紧耦合(p5)。
- 演进:System Prompt 层面的"解耦策略"——**尽量固化 System Prompt,将动态、具体的任务要求剥离出来**。System Prompt 只保留最底层、最通用的系统级指令与基本行为规范、工具信息、Skill 使用方式等"固定"部分,保持极度稳定;任务要求、领域知识、人设规范等"动态内容"拆解存储到外部文件系统,通过**渐进式披露(Progressive Disclosure)**读取加载(p5)。
- 动态性的两个方向(p5-6):
  - **Skill 层面的沉淀**:把执行某项任务的方法论、步骤要求、领域约束沉淀为独立 Markdown 文件(如 SKILL.md),构成 Agent 的"技能库/要求库",执行任务时动态渐进式加载。
  - **配置文件存储**:人设定义、用户偏好、搜索规则等通用规范存入 USER.md、SOUL.md、CLAUDE.md、AGENTS.md 等配置文件,同样渐进式加载,实现 Prompt 内容的模块化管理。
- 本质:从"单体大 System Prompt"到"System Prompt + 渐进式加载上下文文件",是**上下文的组织形式发生了变化**,实现"动静分离":System Prompt 更纯粹稳定,易变业务逻辑/领域知识用结构化 Markdown 灵活加载(p6)。

### 2. Planning:思维链 → 复杂长程任务(p6-7)

- 早期:Lilian Weng 文中 Planning 概念很新鲜,许多人坚信"会对任务做 Planning 的 Agent 才是真正意义上的 Agent";但在基座模型不够强的时代,Planning 实现朴素,主要依赖模型原生 CoT 能力(如 "Let's think step by step" 提示词)做线性、串行逻辑推导;简单任务尚可,复杂场景易陷入逻辑断层或死循环(p6)。
- 现在(随 Reasoning 能力显著增强)具备三种更高级能力(p6-7):
  1. **复杂问题的结构化分解**:主动将宏大模糊目标拆为可执行子任务(Sub-tasks),生成结构化 Todo List。
  2. **多步协同与长程推理**:按任务列表有序执行,执行中动态调整计划,可处理极长上下文依赖的复杂任务并保持逻辑一致性、连贯性。
  3. **子 Agent 的动态构建**:根据子任务需求动态实例化/调用特定子 Agent 专项解决某环节,从"单体思考"到"协同作战"。
- 核心驱动力:**底层基座模型推理能力升级**——模型在逻辑推理、长文本理解、复杂指令遵循上越来越强,Planning 模块从"提示词技巧"演变成真正的"智能决策中枢"(p7)。

### 3. Memory:检索增强 → 文件系统化(p7-8)

- 早期(Lilian Weng 经典架构):Memory 分短期记忆(对话上下文,含 System Prompt、历史 User/Assistant 回复)与长期记忆(外部知识库,通常 RAG 从向量数据库检索文档/知识片段注入模型)(p7)。
- **短期记忆**:核心挑战从"存储"转向"管理"与"压缩"。Context Window 有限且成本敏感;从 OpenClaw、Hermes 等最佳实践看,上下文不再是简单堆砌历史对话,而是引入多种记忆压缩策略(p7-8):
  - 阈值控制:基于固定 token 数或动态语义密度阈值触发压缩;
  - 结构化摘要:对中间过程对话做 Summary 提炼,同时保留头尾关键指令与最终结论,确保核心意图不丢失;
  - 重点提取:从冗长对话流提取关键事实/状态变化,剔除噪音。
  - 效果:短期记忆更精炼、高密度,显著提升模型长对话注意力集中度。
- **长期记忆**:变化更大,从"向量数据库主导"向"文件系统主导"回归,细分两个子方向(p8):
  - **事项型记忆(Episodic Memory)**:用户偏好、历史行为、每日待办等动态"事实",OpenClaw、Hermes Agent 等框架倾向用文件系统记录,如生成 MEMORY.md 或每日 Memory 日志文件,以结构化 Markdown 存储关键事件;比向量检索更可控、更易读,Agent 可直接读取并理解时间序列上的状态变化。
  - **知识型记忆(Semantic Memory)**:随 Karpathy 等提出的 LLM-Wiki、GBrain 这类本地化知识库理念普及,传统纯 RAG 方案正被更灵活的**本地文件系统 + Obsidian 等笔记工具**补充甚至替代,Agent 直接访问组织良好的 Markdown 知识库,非常适合个人知识库。
  - 限制与混合方案:企业级海量知识场景下,仅"文件系统即记忆"+ grep 类关键词检索容易不准确;需搭配 **QMD 或 SQLite 等轻量向量化检索机制**,更复杂场景需企业级向量检索——既保留 RAG 海量知识优势,又赋予开发者通过目录结构、标签、链接显式组织知识的能力,使知识召回更精准、可解释(p8)。
- 本质:从纯向量文本检索走向"**文件系统化的沉淀 + 向量检索混合管理**";短期压缩与长期沉淀都在追求记忆效果、可读性与效率的均衡(p8)。

### 4. Tools:Function Call → CLI / Script(作者认为变化最大的部分)(p8-10)

- 早期主流范式 **Function Call**:针对业务场景把系统能力封装成标准 API 注册为模型可调函数。痛点:**极高的开发与维护成本**——大量系统/数据源没有现成 API,团队需投入大量精力"补全" API,费时费力,且工具数量膨胀后 API Schema 管理极复杂(p9)。
- **MCP(Model Context Protocol)**:在协议层面优化了工具注册与发现机制,实现"一次注册,自动暴露",但本质仍停留在接口标准化层面,未从根本上改变工具调用的底层逻辑(p9)。
- 真正的范式转移:**CLI 命令行原生化 + Script 脚本化**(p9-10)。
- CLI 的特点(p9):
  - **零样本学习优势**:grep、cat、vim 等 Linux/Unix 命令对人类是高门槛,对大模型却是预训练数据中海量代码与技术文档的一部分,属"先天知识";模型通过 CLI 操作文件系统/网络无需额外定义复杂 API Schema(名称、描述、参数类型),节省巨大的 token 空间和调试成本。
  - **可扩展性与自解释性**:未见过的第三方 CLI 工具,只要遵循标准 Linux/Unix 规范(如支持 --help),模型可运行时查询帮助文档、自主理解参数用法并执行调用——"按需查询、即时学习",契合上下文工程的渐进式加载理念。
  - **Skill 集成**:新的第三方 CLI 工具可通过 Skill 包装,在描述文件中提供安装指南和使用示例。
- Script 脚本(Agent Skills 体系中 Resources 形态)成为工具承载主流模式,Python 或其他语言的工具逻辑封装为独立脚本文件,灵活性强(p10):
  1. **本地与远程的统一**:既可直接执行本地命令(文件操作、环境配置),也可内部封装对远程 API 的调用;
  2. **协议的黑盒化**:复杂的 API 鉴权、参数拼接等细节隐藏在脚本内部,Agent 只需关注"调用哪个脚本"与"传入什么核心参数",极大降低模型理解门槛。这也是为什么安装一个 Skill 往往就能赋予 Agent 处理复杂任务的能力——Skill 不仅包含 Prompt 指引,更内置可执行工具脚本。
- 本质:从 Function Call 到 CLI + Script,核心是从"**人为适配模型**"转向"**利用模型原生能力**"——不再为每个操作编写专用 API 接口,而是利用模型预训练积累的通用计算机操作知识(CLI)和代码执行能力(Script),构建更轻量、灵活、易扩展的工具生态(p10)。

### 5. Workflow:刚性编排 → 动态混合封装(p10-11)

- 早期:基础模型指令遵循与逻辑稳定性较弱,依赖显式、硬编码 Workflow 保障执行,类似传统"状态机"或"流水线 Pipeline",把复杂任务拆为严格固定的第一步、第二步、第三步,强制模型按部就班;牺牲灵活性,但当时是确保 Agent 不"跑偏"、不掉链子的必要手段(p10)。
- 问题:运行非常"机械化",无论外部环境如何变化仍死板地严格遵循步骤,**无法根据实际情况做出动态调整**(p10)。
- 重构方向:**从"刚性的流程编排"转向"动态的 Skill 封装与混合架构"**,两部分(p11):
  - **逻辑内聚化**:原本分散在 Workflow 引擎中的步骤定义、约束条件、核心判断逻辑,直接写入 Skill 的 Markdown 描述文件(如 SKILL.md);模型读 Skills 文档即可理解任务完整链路。
  - **执行脚本化**:需要精确控制的环节,不再依赖外部工作流引擎的状态跳转,而是通过 Skill 关联 Resources 的 Script 脚本进行代码级编排和控制——一个复杂业务流程可被打包成一个独立、可复用的 Skill。
- 新挑战:**可控性与稳定性的博弈**。纯 Skill 驱动赋予 Agent 更高自主性,但在极端复杂或容错率极低场景,模型仍可能理解偏差或执行跳跃,结果不可控;刚性 Workflow 虽笨重,却提供确定性边界(p11)。
- 当前处于**新旧技术交叉融合的过渡期**,企业级落地倾向混合架构(p11):
  - 成熟、标准化的子任务封装为 Skills(Markdown 维护逻辑,灵活易用);
  - Workflow 的固定运行部分其实可全部转换为 Script,但代码可读性很多时候不如 Workflow;因此关键的、稳定性要求极高的主干流程仍保留为 Workflow,或将特定 Workflow 封装为特殊 Tool 供 Agent 调用,是现阶段较好的办法。
  - "**Skill 为主,Workflow 为辅/兜底**"的策略,既利用新技术红利又保留确定性,是当前平衡开发效率与运行稳定性的最佳实践。

### 6. Environment:无状态 → 运行时环境(p11-12)

- 早期 Agent 的工具调用、子 Agent 调用都是无状态的,几乎不需要"运行环境"。随着引入文件系统操作、代码执行等能力,Agent 不再只是"问答机器",而是需要持久化存储、文件读写和状态管理的"数字员工",必须拥有专属 **Workspace(工作空间)**:安全地读取配置、写入日志、生成中间文件,管理其 Skill 和 Memory 数据(p12)。
- Runtime 环境两种形态(按应用场景与安全要求)(p12):
  - **本地个人电脑(Local Desktop)**:便利性、灵活性极高,可直接操作本地文件系统、应用和网络,实现"整理桌面文件""自动化办公流程"等贴近个人生活的复杂任务;OpenClaw 最早就是基于个人电脑操作而"火"起来的。但缺乏严格隔离机制,操作失误可能导致重要数据丢失或系统配置混乱,通常需引入更严格的用户确认机制或权限控制。
  - **沙箱环境(Sandbox / Cloud Server)**:企业级生产环境的主流选择。通过 Docker、Kubernetes 等容器化技术构建隔离沙箱,所有操作被限制在特定虚拟文件系统内;即使执行破坏性命令也不影响宿主机或其他服务,提供安全边界与资源管控,确保不可预测行为下系统整体稳定。

## 总结要点(p12-13)

六维度演变一览(原文总结):
- **Prompt**:单体"小作文" → 解耦的上下文工程;
- **Planning**:线性 CoT 思维链 → 复杂的长程任务拆解;
- **Memory**:传统前置向量检索 → 文件系统化 + 向量检索混合架构;
- **Tools**:高成本 API 封装 → 回归原生 CLI 与脚本交互;
- **Workflow**:刚性外部编排 → 内化为灵活的 Agent Skills 封装;
- **Environment**:无状态调用 → 有状态的隔离运行时 Runtime 系统环境。

不变的核心目标:**在保证安全、可控的前提下,最大化释放模型的推理与执行潜力,让 Agent 真正成为能够解决复杂现实问题的得力助手**(p13)。

给从业者的建议:理解演变背后的逻辑比掌握具体工具更重要——模型会继续升级、工具会继续变化、框架会持续更新,但"**通过工程化手段构建确定性,以承载模型不确定性**"的核心思想将是未来很长一段时间内构建高质量 Agent 的基石(p13)。作者自述本文为个人技术探索心路历程、一家之言(p13)。
