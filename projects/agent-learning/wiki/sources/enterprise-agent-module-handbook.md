## 文档定位与核心论点

- 目标读者读完应能回答三个问题(p2):
  1. 为什么不能只把大模型接到聊天框里就叫业务 Agent;
  2. 架构里每个模块到底解决什么业务问题;
  3. 换一个新业务场景,产品经理怎么把业务套进这套 Agent 模板。
- 普通聊天机器人 = 理解→生成→回答;企业业务 Agent = 识别任务→校验参数→调用 Skill。关键不是"回答得像不像",而是"能不能按流程稳定推进"(p2,图1)。
- 普通聊天可以"说错了再改",企业业务流程不行:接入凭证、上线发布、数据结论、规则更新都需要稳定判断、权限边界和可追溯记录(p4)。

## Agent 演进史与混合架构选型(p3)

四阶段演进(图2):

| 阶段 | 特征 | 适用 |
|---|---|---|
| 早期 Agent | 聊天 + 工具调用 | 短链路任务 |
| 工作流 Agent | 流程编排 | 企业稳定流程 |
| 自主 Agent | 规划 + 执行 | 开放复杂任务 |
| 自进化 Agent | Skill + Memory,沉淀经验资产 | 经验沉淀 |

核心启发:**对企业业务,不是越自主越好,而是要把模型能力放进可控流程里**(p3)。

为什么选混合架构——业务特征→需求→模块的映射(p3,图3):

| 业务特征 | Agent 需要什么 | 对应模块 |
|---|---|---|
| 流程顺序稳定 | 不要每次重想流程 | Workflow |
| 字段规则明确 | 判断可校验 | Schema + Validator |
| 问题高度重复 | 复用专家经验 | Skill + 知识库 |
| 任务多轮推进 | 知道走到哪一步 | Task State |
| 凭证上线敏感 | 审计与人工确认 | Audit + 人工审核 |

一句话:LLM 负责理解表达,流程、校验、状态、审计和人工确认负责稳定可控(p3)。

## 主链路:业务任务如何被接住(p4,图4)

业务主流程:**用户输入 → 会话路由 → 需求转化 → Schema 校验 → 调用 Skill → 输出确认**;LLM 作为"增强层"(识别意图、抽取字段、追问润色、输出润色)挂在多个节点上,**LLM 是增强点,不是唯一决策者**。支撑与边界四件套:Task State(记住进度)、Memory(沉淀经验)、Audit(记录风险)、人工确认(高风险放行)。

### LLM 放在哪里:适合做什么 vs 不应独自做什么(p4-5)

| 位置 | LLM 适合做什么 | 不应该 LLM 独自做什么 |
|---|---|---|
| Session Router | 判断这句话像闲聊、补充信息还是新任务 | 不能因为模型觉得像业务就强行写入任务 |
| 需求识别与转化 | 把自然语言抽取成任务类型、字段、问题类型 | 不能绕过 Schema 自行认定可执行 |
| 追问策略 | 把阻塞项问得清楚、少问废话 | 不能为补全信息索要敏感凭证 |
| 知识问答 | 结合检索片段生成容易理解的回答 | 不能在无依据时编造接口规则 |
| 输出润色 | 把排查建议、分析结论写得更像人话 | 不能替代人工审批和生产操作 |

产品经理对外口径(p5):"我们不是让大模型直接接管业务,而是让大模型嵌入到可控流程里:它负责听懂、抽取、解释和表达;能不能执行、是否要追问、是否涉及安全边界,由 Schema、规则、权限和人工审核共同决定。"

## 十二个模块逐一解析(p5-7)

这部分是手册核心:理解成一套业务 Agent 的标准工作分工——谁接任务、谁判断、谁追问、谁调用能力、谁保存状态、谁沉淀经验、谁管风险。

1. **入口层**(p5):用户和 Agent 接触的地方——WebUI(产品/商务/技术支持用)、CLI(技术/内测)、未来的本地接口或服务接口。价值不是炫技,而是把业务需求自然地接进 Agent。例:商务输入"某三方要接 API,问我们要 appid 和 appsecret"。
2. **Session Router**(p5):会话分诊台——判断这句话是闲聊、新任务、补充上一任务的信息,还是修改上一任务。解决的是:避免把"好的""这个不对""换成昨天的数据"都强行当成新的 API 接入问题。口诀:入口层负责接住人,Session Router 负责先分清话;一个让任务进得来,一个避免任务进错门。
3. **需求识别与转化**(p5):把自然语言变成结构化业务任务。例:"对方说签名一直失败,帮忙看下" → `request_type = signature_debug`;问题类型 = 签名/鉴权排查;需要补充 = 接口路径、appid、错误码、请求时间、签名原始参数;安全提醒 = 不要提供 appsecret。LLM 适合理解表达、识别意图、抽取参数、判断归类,但**只帮助理解,不负责最终放行**。
4. **Schema Builder + Validator**(p5-6):Schema 是业务任务单模板,Validator 是检查任务单是否合格的检查员。校验结果三类:`executable`(可继续执行)/ `needs_clarification`(需要追问)/ `blocked`(触发规则或安全边界,不能继续)。例:用户发了 appsecret 或要求 Agent 生成 appsecret → 进入 `security_flags` 而非继续对话。核心理解:**LLM 可以帮你填表,但不能自己决定表是否合格;表是否合格,要靠 Schema 和业务规则校验**(p6)。
5. **Clarification Policy**(p6):追问策略——"缺什么就只问什么"。很多 AI 助手不好用是因为一次问一堆问题,让业务人员烦,还把敏感信息带进对话。追问要区分四类问题:**缺失**(必须字段没有)、**模糊**(表达不清楚)、**冲突**(用户说法和规则冲突)、**非法**(触发权限或安全边界)。可以让 LLM 做话术润色,但问哪些字段由 Schema 和规则决定。
6. **Skill Router + SkillLoader**(p6):Skill Router 像派单系统——新接入→API 接入流程 Skill,字段问题→API 文档问答 Skill,签名失败→鉴权与签名排查 Skill,回调异常→回调排查 Skill,续租失败→续租规则 Skill,单量分析→单量分析 Skill。SkillLoader 是架构增强点:不是一开始把所有 Skill 全塞给模型,而是**分阶段加载:Advertise(先知道有哪些 Skill)→ Load(命中后读取完整 Skill)→ Read(需要时读取参考资料)→ Run(必要时运行脚本或工具)**。类比:Skill 是业务专家能力包,Skill Router 是把问题交给哪个专家。
7. **知识库与文档检索**(p6):为了避免模型瞎编。API 业务资料包括 API 文档、FAQ、SDK 文档、接入流程、错误码说明、续租规则、货架/号池规则。回答应先检索文档,再基于文档回答,而不是让模型凭经验猜;无命中文档依据时应明确说明未找到。解决:回答可追溯、口径一致。
8. **Tool Governance**(p6-7):未来接 BI 查询、日志查询、后台配置、凭证系统、订单系统等真实系统后,工具不再只是回答问题,而是能影响真实业务结果,必须受控。要管理:这个工具能不能调用、谁有权限调用、是否只允许 dry-run、是否需要人工确认、是否记录审计日志。一期可以很轻量:只做样例脚本、dry-run、权限说明,不接真实后台;接真实系统前再加强治理。边界提醒:越接近真实后台、真实凭证、真实数据,越需要权限、审计和人工确认(p7)。
9. **Task State**(p7):企业业务 Agent 和普通聊天机器人最大的区别之一。要记住任务状态:需求确认中、资料待补充、凭证待生成、接入包待发放、沙箱联调中、上线前检查中、已上线、上线后监测中、暂停/阻塞等。没有 Task State,用户说"资料补齐了,下一步呢",Agent 不知道之前走到哪一步。解决:**多轮推进,而不是每轮从零开始**。
10. **Memory**(p7):不是简单记住聊天记录,而是记录识别错误、用户纠正、真实联调案例、重复问题、高频 FAQ 和 Skill 优化建议,目标是让 Agent 越用越懂业务。例:Agent 把"续租失败"误判成"订单回调问题",用户纠正后这条纠错应进入 Memory,后续变成优化识别规则、补充 few-shot 示例、更新续租 Skill、加入测试集。解决:**把个体经验逐步变成组织能力**。
11. **Runtime Logs / Audit**(p7):企业 Agent 的安全底座。记录:用户输入摘要、识别成什么任务、Schema 校验结果、缺了哪些字段、命中哪个 Skill、是否调用了 LLM、是否触发安全边界、最终输出摘要。但**不能记录敏感信息**(appsecret、真实密钥、完整敏感凭证)。解决:出了问题能复盘,有风险能审计,同时不把敏感信息沉淀到普通日志里。
12. **人工审核发布**(p7):最后一道边界。Agent 可以生成建议、整理清单、输出草稿、给排查步骤、生成 Skill 优化建议,但**不能自动生成 appsecret、发放凭证、改后台配置、正式上线、自动更新生产 Skill**。这层解决企业 Agent 可控落地而非失控自动化。对内讲解必须讲清:Agent 是帮人推进流程,不是绕过人和权限系统。

一句话记忆(p7):**Task State 管推进,Memory 管进化,Audit 管风险,人工审核管最终放行。**

## 一期范围:哪些必须做,哪些后续增强(p8)

| 模块 | 一期建议 | 原因 |
|---|---|---|
| 入口层 | 必须有 | 用户需要一个发起任务的位置 |
| Session Router | 必须有轻量版 | 避免闲聊、补充信息、新任务混在一起 |
| 需求识别与转化 | 必须有,可接 LLM | 自然语言必须转成业务任务 |
| Schema + Validator | 必须有 | 判断能不能继续执行,不能只靠模型感觉 |
| 追问策略 | 必须有 | 只问阻塞项,减少用户负担 |
| Skill Router + Skill | 必须有 | 不同问题要走不同业务能力 |
| 知识库检索 | 一期轻量做 | 先用文档摘要和关键词检索,不必一开始做复杂向量库 |
| Task State | 必须有轻量版 | 支持多轮推进和草稿修改 |
| Memory | 一期轻量做 | 先记录误判、纠错、重复问题 |
| Tool Governance | 接真实系统前加强 | 涉及后台、BI、凭证时必须有权限和审计 |
| Audit + 人工审核 | 必须有 | 这是企业场景的安全边界 |

总原则:核心链路先跑通,检索、日志、Memory 先轻量,真实工具和平台化能力后续再接(p8)。

## 产品经理落地方法论(p8-9)

不要从"我要做一个万能 Agent"开始,而是先选一个明确业务闭环,拆成 Workflow、Schema、Skill、State 和验收场景。落地五步(图5):**业务流程(选高频闭环)→ 字段规则(定义 Schema)→ 追问策略(只问阻塞项)→ 业务能力(沉淀 Skill)→ 验收沉淀(测试与 Memory)**。产品经理的核心工作:把流程、字段、规则、验收先定义清楚;LLM 负责增强理解和表达。

选第一期场景的四条判断标准(p9):
1. **高频**:这个问题每天或每周反复出现;
2. **规则明确**:不是完全靠人情判断,而是有文档、SOP、字段、审批边界;
3. **可人工确认**:Agent 先出草稿或建议,人来最终确认;
4. **能做验收**:可以拿真实问题或历史案例测试 Agent 是否答对。

## 示例业务:API 接入与分析 Agent(p9)

API 接入场景天然适合第一批业务 Agent:固定流程、固定材料、固定文档、固定排查套路、明确安全边界;三方常见问题大量重复,内部同学也需要持续做单量整理和上线后监控。

三条任务链分别跑流程,但共用底座 Schema / Skill / Task State / Memory / Audit(图4):
- **新 API 接入**:需求确认 → 资料补齐 → 凭证准备 → 上线检查;
- **联调问题**:识别问题 → 补齐日志 → 排查建议 → 沉淀案例;
- **单量分析**:确认口径 → 趋势分析 → 异常判断 → 输出周报。

安全边界:appsecret 不进模型上下文;凭证、上线、Skill 更新必须人工确认(p9)。

API 业务一期 Agent 能力范围(p9):
- 接住新接入诉求,判断资料是否齐全,输出接入准备清单;
- 回答 API 文档、字段、SDK、货架、号池、续租相关常见问题;
- 针对签名失败、回调异常、错误码等问题,追问关键日志并给出排查路径;
- 基于样例数据做单量趋势、成功率、失败率和异常波动分析;
- 记录重复问题、误判和真实联调案例,形成后续 Skill 优化建议。

## 端到端案例:三方说"签名失败"(p10)

| 阶段 | Agent 做什么 | 产品经理要关注什么 |
|---|---|---|
| 入口层 | 接收"签名失败"这句话 | 入口是否足够方便,是否适合内部同学使用 |
| 会话路由 | 判断这是新问题还是延续上次联调 | 避免把无关上下文混进来 |
| 需求识别 | 识别为 `signature_debug` | 是否能识别口语化表达 |
| Schema 校验 | 发现缺接口路径、appid、签名原始参数、错误码、日志时间 | 缺的是不是阻塞项 |
| 追问策略 | 只追问上述关键字段,并提醒不要提供 appsecret | 追问是否简洁、安全 |
| Skill 路由 | 命中鉴权与签名排查 Skill | 排查步骤是否来自 SOP/文档 |
| 输出结果 | 给出排查建议和下一步动作 | 是否能让技术支持直接拿去用 |
| Memory/Audit | 记录本次问题类型和安全提醒 | 是否能沉淀为案例,是否保护敏感信息 |

关键安全口径(p10):**appsecret 不进模型上下文、不在对话中复述、不写入普通日志;查看、重置、发放必须走有权限和审计的安全通道。**

## 一期验收标准(p11)

验收不要只看"回答好不好看",要看是否稳定走完正确链路,尤其是需求识别、追问、Skill 路由、安全边界和人工确认。

| 验收场景 | 应该看到的结果 |
|---|---|
| 三方说"给一下 appid 和 appsecret" | 识别为新接入或凭证相关请求;先检查资料;不生成、不复述 appsecret |
| 三方说"签名失败" | 追问接口路径、appid、签名参数、错误码、日志时间;提醒不要发 appsecret |
| 三方说"没有收到回调" | 追问回调地址、订单号、触发动作、服务端日志,并进入回调排查 Skill |
| 三方问"某字段什么意思" | 命中文档问答或参数解释 Skill,并说明依据来源 |
| 三方问"续租为什么失败" | 按续租规则排查投诉、黑名单、时间冲突、未开始续租单等原因 |
| 内部问"某渠道近 7 天单量" | 确认渠道、时间、游戏范围、指标口径,输出趋势、异常点和建议 |

验收口径(p11):**初版不追求"无所不答",而是追求在核心场景里达到 80 分:识别稳定、追问克制、输出有依据、安全边界清楚。**

## 推荐落地路线(七步,p11)

从"搭一个聊天助手"调整为"搭一个高频业务流程 Agent",重视流程、Schema、Skill、状态、沉淀和审核,而非一开始追求大而全:
1. 选一个业务闭环,不做万能入口;
2. 写清 Workflow(任务从发起到结束的主流程);
3. 定义 Schema,只保留会影响判断、追问、阻断的字段;
4. 整理一期 Skill,先覆盖高频 SOP、文档问答、排查、分析;
5. 接入 LLM 做理解和表达增强,但保留规则兜底和安全边界;
6. 做验收集,用真实问题压测识别、追问、路由和安全;
7. 把误判、纠错、重复问题进入 Memory,再由人工审核更新 Skill。

扩展时机(p11):当一个业务 Agent 能稳定处理核心场景,且日志、Memory、Skill 更新机制跑通后,再抽象成通用底座;此时新业务只需替换 Workflow、Schema、Skill、知识库和验收集,不用重新发明整个 Agent 架构。
