三方标准号池方案

需求背景

租号玩与三方渠道存在合作关系,我们会供给游戏账号给到三方平台进行出租盈利。但目前我们提供
给三方的账号形式不统一,每家获取的方式都需要单独维护,整体比较麻烦。

现有三方货架同步规则

顺网端游、⻜⻔租号端游没有号池,直接从租号玩大号池中筛选拉取数据;

GG租号端游、手游和顺网手游都是号池的。通过筛选规则同步到号池后,三方从号池拉取数据。

需求简介

通过打造三方的标准号池,让三方合作的渠道都统一去这个标准号池中取账号数据。

需求详述

号池规则定义

1. 通用货架筛选规则

🍩 根据下方各类的游戏要求及定义,从租号玩大号池中筛选符合要求的货架进入三方标准号
池,满足数量和品质需求。

在符合货架品质要求的前提下,整体按照热度推送,在各自的价格区间中,判断货架品质并
获取货架,同时根据货架归属的号主来判断是否可以进行、推送。

2.3
货架取用定义

• 目前⻜⻔通过搜筛获取货架时,专属自营货架数量占比30%,租号玩正常货架占比70%,且专属货

• 专属货架按照设定比例进行固定展示,向下不兜底,如果该游戏无专属货架数据时,则全部展示租

4. 标准号池推送定义

2.1
三方请求/推送定义

• 大数据通过请求接口中的三方ID作为唯一标识,通过各自接口中的货架筛选规则从标准号池推送货
架数据至三方。

2.3
标准号池货架剔除策略

标准号池的货架推送按照我们平台定义的规则为主,三方在接收货架后可以定义自己的删除策略(尽可
能地让三方少做事情,提升我们的推送货架品质)。

• 三单连续撤单

• 货架七天状态无变化(七天内没有出租状态的)的数据加入大数据删除库,不再向号池推送;后续推
送策略再次筛选到此货架后,向标准号池推送时,对应货架从待删除库中剔除,标准号池继续接收
此货架。

号池线上测试调度程序&正式环境调度程序

• 货架调度机制

## Page 11

◦ 通过筛选规则筛选出测试货架时,根据当次测试的要求(上号方式/数量/游戏名词等),大数据侧
修改/增加筛选条件去选定符合条件的货架,为对应货架打上对应的测试货架标识。

◦ PHP接口在从号池获取货架池时,默认情况下所有接口通过筛选条件不获取带有测试标识的货
架,在某个三方端口进行测试时,修改对应三方的php接口的请求条件,加上获取测试货架的
字段后,在原有三方请求的基础上,加入带有测试标识的货架。

• 测试调度程序和正式环境调度程序需要满足测试和正式互不影响。

• 某个三方进行测试时,不影响标准号池同步拉取数据同时支持其他三方正常获取标准号池中的货架
数据。

• 测试货架应该受到的唯一的影响点是,被平台租赁走,导致状态变为租赁中,会在标准号池同步货
架时被剔除。

号池预警同步机制

遇到的问题:

目前双方接口读取数量设置的没问题,但接口本身的性能会存在不稳定性,导致会出现无法读取那么
多数据的情况。

• 预警检测指标在⻜书群播报

◦ 大号池向标准号池推送的数据量检测,检测重点游戏(穿越火线)的推送量,设置95%的阈值作为
告警标准。
